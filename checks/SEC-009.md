# SEC-009 — Session-ID Cryptographic Binding

## Description

In statusbehafteten MCP-Setups mit Streamable HTTP / SSE wird jeder Verbindung eine `Mcp-Session-Id` zugewiesen. Diese ID identifiziert den Sitzungszustand auf Server-Seite und wird bei jedem Folge-Request mitgesendet.

**Angriffsmuster (Session Hijacking):**

1. Server generiert vorhersagbare Session-IDs (z.B. inkrementell oder mit schwachem PRNG)
2. Angreifer rät / erschöpft Session-ID-Raum
3. Angreifer sendet bösartiges Event (Tool-Call) mit der erratenen Session-ID
4. Server führt Operation im Kontext des Opfers aus

**Verschärfung in Cloud-Architekturen:** Bei Shared-State-Setups (z.B. Redis als Session-Store, siehe SCALE-002) liegen alle Sessions in einer gemeinsamen Queue. Eine einzige geleakte Session-ID kompromittiert nicht nur die User-Verbindung, sondern erlaubt auch Event-Injection in laufende Konversationen.

**Mitigation in zwei Schritten:**

1. **Kryptografisch sichere Generierung:** UUIDv4 oder `secrets.token_urlsafe(32)` — niemals `random.random()` oder Timestamps.
2. **Binding an User-Identität:** Session-ID wird kombiniert mit der `user_id` aus dem **validierten** OAuth-Token (nicht aus Client-Headern). Format: `<user_id>:<session_id>`. Bei jedem Request wird geprüft, ob die Session-ID zur authentifizierten User-Identität gehört.

## Verification

### Modus 1: code_review (kryptografisch sichere Generierung)

```bash
# Suche nach Session-ID-Generierung
grep -rE "session_id|sessionId|mcp_session" src/ | grep -E "=|generate|create"
grep -rE "uuid\.|secrets\.|crypto\." src/
```

**Pass-Pattern (Python):**

```python
import secrets

def new_session_id() -> str:
    # 256 Bit Entropie, URL-safe Base64
    return secrets.token_urlsafe(32)
```

**Pass-Pattern (TypeScript / Node):**

```typescript
import crypto from "node:crypto";

export function newSessionId(): string {
  return crypto.randomUUID();  // RFC 4122 v4, kryptografisch sicher
}
```

**Fail-Pattern:**

```python
# FAIL: Math.random / time-based
session_id = str(int(time.time())) + str(random.randint(0, 99999))

# FAIL: hashlib.md5 mit Sequenz
session_id = hashlib.md5(f"session-{counter}".encode()).hexdigest()
```

### Modus 2: code_review (Binding an validated user_id)

```bash
# Suche nach Session-Binding-Logik
grep -rE "user.*session|sub.*session|session.*sub" src/
grep -rE "validate_session|verify_session" src/
```

**Pass-Pattern:**

```python
from authlib.jose import jwt

class SessionStore:
    def create(self, user_id: str) -> str:
        session_id = secrets.token_urlsafe(32)
        bound_key = f"{user_id}:{session_id}"
        await self.redis.setex(
            f"mcp:session:{bound_key}",
            ttl=3600,
            value=json.dumps({"user_id": user_id, "created_at": time.time()}),
        )
        # Token-Form für Client: signed JWT, nicht raw bound_key
        return self.signer.dumps(bound_key)

    def validate(self, signed_token: str, claimed_user_id: str) -> dict:
        try:
            bound_key = self.signer.loads(signed_token, max_age=3600)
        except BadSignature:
            raise PermissionError("Invalid session token")
        stored_user_id, _, _session_id = bound_key.partition(":")
        if stored_user_id != claimed_user_id:
            raise PermissionError("Session does not belong to this user")
        data = await self.redis.get(f"mcp:session:{bound_key}")
        if not data:
            raise PermissionError("Session expired")
        return json.loads(data)


# Verwendung im Request-Handler
@mcp.middleware
async def session_middleware(request, call_next):
    auth_header = request.headers.get("Authorization", "")
    claims = verify_client_token(auth_header)  # aus SEC-002
    user_id = claims["sub"]

    session_token = request.headers.get("Mcp-Session-Id")
    if session_token:
        session_data = await session_store.validate(session_token, user_id)
        request.state.session = session_data
    else:
        # Neue Session erzeugen
        new_token = await session_store.create(user_id)
        response = await call_next(request)
        response.headers["Mcp-Session-Id"] = new_token
        return response

    return await call_next(request)
```

**Fail-Pattern:**

```python
# FAIL: User-ID kommt aus Client-Header, nicht aus validiertem Token
session_id = request.headers["X-User-Id"] + ":" + raw_session

# FAIL: Kein Binding-Check, nur Existenz-Check
if redis.exists(f"session:{session_id}"):
    proceed()  # ← jeder mit der ID darf weiter
```

### Modus 3: runtime_test (Session-Hijacking-Versuch)

```bash
# 1. Legitime Session als User A erzeugen
SESSION_A=$(curl -s -X POST https://server/mcp/init \
  -H "Authorization: Bearer $TOKEN_USER_A" \
  -d '{}' | jq -r '.session_id')

# 2. User B versucht, mit User-As-Session-ID zu agieren
curl -X POST https://server/mcp/call \
  -H "Authorization: Bearer $TOKEN_USER_B" \
  -H "Mcp-Session-Id: $SESSION_A" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' \
  -w "\nHTTP %{http_code}\n"
# Erwartet: HTTP 401 oder 403 — Session gehört nicht zu User B
```

## Pass Criteria

- [ ] Session-ID-Generierung verwendet `secrets.token_urlsafe()` / `crypto.randomUUID()` (≥128 Bit Entropie)
- [ ] User-ID kommt aus **validiertem** OAuth-Token (`sub`-Claim), nicht aus Client-Headern
- [ ] Session-ID ist kryptografisch an User-ID gebunden (z.B. signiertes JWT mit `user_id`+`session_id` als Payload)
- [ ] Bei Mismatch (Session gehört zu anderem User): HTTP 401/403
- [ ] Session-TTL ist explizit gesetzt (z.B. 1h)
- [ ] Sessions werden bei Logout serverseitig invalidiert (nicht nur Client-Cookie löschen)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| `random.randint()` oder `time.time()` als Session-ID | Vorhersagbar, Brute-Force möglich |
| Session-ID ohne User-Binding | Geleakte ID = vollständiger Account-Zugriff |
| User-ID aus Client-Header für Binding | Header-Spoofing umgeht Binding |
| Kein TTL — Sessions leben ewig | Single-leak = Permanent-Compromise |
| Logout löscht nur Cookie, nicht Server-State | Token bleibt nach Logout gültig |
| Session-ID in URL-Query-Params | leak via Referer / Logs |

## Remediation

Volles Pattern oben. Zusätzlich:

### Session-Rotation bei Privilege-Wechsel

```python
@mcp.tool()
async def elevate_to_admin(ctx: Context, justification: str):
    # Bei Privilege-Wechsel neue Session-ID generieren
    new_token = await session_store.create(ctx.user_id, scopes=["admin"])
    ctx.set_response_header("Mcp-Session-Id", new_token)
    # Alte Session invalidieren
    await session_store.invalidate(ctx.session_token)
```

### Logout-Endpoint mit Server-Side-Invalidation

```python
@app.post("/mcp/logout")
async def logout(request: Request):
    session_token = request.headers.get("Mcp-Session-Id")
    if session_token:
        await session_store.invalidate(session_token)
    return Response(status_code=204)
```
