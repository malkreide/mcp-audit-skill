# SDK-004 — CORS Mcp-Session-Id Exposure

## Description

Bei Streamable HTTP / SSE läuft die MCP-Kommunikation über Cross-Origin-Requests, wenn der Client (Browser-basiert) auf einer anderen Domain als der Server hostet. Der Server gibt nach `init` einen `Mcp-Session-Id`-Header in der Response zurück — diesen muss der Browser an Folge-Requests anhängen können.

Das Problem: Browser blockieren standardmässig den Zugriff auf Custom-Response-Headers via JavaScript (CORS-Spezifikation). Damit der Client den `Mcp-Session-Id`-Header lesen kann, muss der Server ihn explizit über `Access-Control-Expose-Headers: Mcp-Session-Id` freigeben.

Wird dieser Header vergessen:
- Browser-Client kann die Session-ID nicht auslesen
- Folge-Requests landen ohne Session-Header
- Server denkt, jeder Request sei eine neue Session
- Stateful-Tools brechen, Konversation kollabiert

Dieses Symptom ist subtil: lokale stdio-Tests funktionieren perfekt, Server-Side-Curl-Tests funktionieren — nur der echte Browser-Use-Case bricht.

## Verification

### Modus 1: code_review (CORS-Middleware-Konfiguration)

```bash
# Suche nach CORS-Middleware in der ASGI-App
grep -rE "CORSMiddleware|cors|allow_origins|expose_headers" src/
```

**Pass-Pattern (Starlette / FastAPI):**

```python
from starlette.applications import Starlette
from starlette.middleware import Middleware
from starlette.middleware.cors import CORSMiddleware

middleware = [
    Middleware(
        CORSMiddleware,
        allow_origins=os.environ.get("ALLOWED_ORIGINS", "").split(","),
        allow_methods=["GET", "POST", "OPTIONS"],
        allow_headers=["Content-Type", "Mcp-Session-Id", "Authorization"],
        expose_headers=["Mcp-Session-Id"],  # ← KRITISCH
        allow_credentials=True,
    ),
]

app = Starlette(
    routes=[Mount("/", app=mcp.streamable_http_app())],
    middleware=middleware,
)
```

**Pass-Pattern (FastMCP direkt):**

```python
mcp = FastMCP(
    "zurich-opendata",
    cors_origins=os.environ.get("ALLOWED_ORIGINS", "").split(","),
    cors_expose_headers=["Mcp-Session-Id"],
)
```

**Fail-Pattern:**

```python
# FAIL: CORS aktiv, aber expose_headers fehlt
middleware = [
    Middleware(
        CORSMiddleware,
        allow_origins=["*"],
        allow_methods=["*"],
        allow_headers=["*"],
        # expose_headers nicht gesetzt → Browser kann Mcp-Session-Id nicht lesen
    ),
]
```

### Modus 2: runtime_test (Live-Verhalten von Browser-Sicht)

```bash
# Cross-Origin-Probe simulieren mit Origin-Header
curl -i -X POST https://server/mcp \
  -H "Origin: https://test-client.example.com" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{}}'
```

**Pass:** Response enthält:
```
Access-Control-Allow-Origin: https://test-client.example.com
Access-Control-Expose-Headers: Mcp-Session-Id
Mcp-Session-Id: <some-id>
```

**Fail:** `Access-Control-Expose-Headers` fehlt oder enthält `Mcp-Session-Id` nicht.

### Modus 3: code_review (Allowed Origins explizit)

`allow_origins=["*"]` ist bei `allow_credentials=True` ein CORS-Spec-Verstoss und gefährlich. Pro Production: explizite Origin-Liste.

```bash
grep -rE "allow_origins" src/
```

**Pass:** Origin-Liste aus Env-Var, in Production keine `*`.

## Pass Criteria

- [ ] CORS-Middleware ist konfiguriert (sofern HTTP/SSE-Transport aktiv)
- [ ] `expose_headers` enthält `Mcp-Session-Id`
- [ ] `allow_headers` enthält ebenfalls `Mcp-Session-Id` (für Folge-Requests)
- [ ] `allow_origins` ist in Production keine Wildcard `*`, sondern explizite Liste
- [ ] Bei Auth: `allow_credentials=True` mit explizitem Origin (nicht Wildcard)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| `expose_headers` nicht gesetzt | Browser-Clients brechen, Server-Side-Tests funktionieren — schwer zu diagnostizieren |
| `allow_origins=["*"]` mit `allow_credentials=True` | CORS-Spec-Verstoss, Browser blockiert |
| `allow_headers=["*"]` ohne explizite Liste | Mcp-Session-Id wird in einigen Browsern nicht durchgereicht |
| CORS-Konfig in Code hardcoded | Verschiedene Environments (dev/staging/prod) brauchen unterschiedliche Origins |

## Remediation

```diff
  from starlette.applications import Starlette
  from starlette.routing import Mount
+ from starlette.middleware import Middleware
+ from starlette.middleware.cors import CORSMiddleware

+ ALLOWED_ORIGINS = [
+     o.strip() for o in os.environ.get("ALLOWED_ORIGINS", "").split(",") if o.strip()
+ ]
+
+ middleware = [
+     Middleware(
+         CORSMiddleware,
+         allow_origins=ALLOWED_ORIGINS,
+         allow_methods=["GET", "POST", "OPTIONS"],
+         allow_headers=["Content-Type", "Mcp-Session-Id", "Authorization"],
+         expose_headers=["Mcp-Session-Id"],
+         allow_credentials=True,
+     ),
+ ]
+
  app = Starlette(
      routes=[Mount("/", app=mcp.streamable_http_app())],
+     middleware=middleware,
  )
```

Plus Umgebungsvariable:

```bash
# .env (production)
ALLOWED_ORIGINS=https://app.schulamt.zh.ch,https://claude.ai
```
