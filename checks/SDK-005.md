# SDK-005 — TypeScript Strict Mode + Zod Runtime Validation

## Description

TypeScript verliert seine Typsicherheit nach der Transpilation zu JavaScript. Externe Eingaben — wie sie von einem probabilistischen LLM generiert werden — können daher nicht durch Compile-Time-Typing geschützt werden. Der Goldstandard im TypeScript-MCP-Ökosystem ist Zod: eine TypeScript-first Schema-Library, die Runtime-Validation **und** Type-Inference liefert.

Ohne Zod (oder vergleichbare Validation): Falsche LLM-Eingaben crashen den Tool-Handler oder verursachen subtile Bugs, die erst in Production sichtbar werden.

## Verification

### Modus 1: config_check (`tsconfig.json`)

```bash
# Strict-Modus muss aktiv sein
cat tsconfig.json | grep -E '"strict"\s*:\s*true'
```

**Pass:** `"strict": true` ist gesetzt.
**Fail:** `"strict": false` oder Eigenschaft fehlt.

### Modus 2: automated (Zod als Dependency)

```bash
cat package.json | grep -E '"zod"\s*:'
```

**Pass:** `"zod": "^3.x"` oder neuer in `dependencies`.
**Fail:** Zod fehlt.

### Modus 3: code_review (Tool-Registrierung)

Tools müssen `inputSchema` als Zod-Objekt definieren, **nicht** als rohes JSON-Schema oder TypeScript-Interface.

```bash
# Suche nach Tool-Registrierungen mit nicht-Zod-Schemas
grep -rB2 "inputSchema" src/ | grep -E "type|interface"
```

**Pass-Pattern:**
```typescript
import { z } from "zod";

server.registerTool({
  name: "getUserInfo",
  description: "...",
  inputSchema: z.object({
    userId: z.string().uuid(),
    includeArchived: z.boolean().optional(),
  }),
  handler: async (args) => { /* args ist hier typsicher */ },
});
```

**Fail-Pattern:**
```typescript
// JSON-Schema direkt
inputSchema: {
  type: "object",
  properties: { userId: { type: "string" } },
  required: ["userId"],
}

// Oder TS-Interface ohne Runtime-Validation
interface Args { userId: string; }
handler: async (args: Args) => { /* keine Validation, vertraut LLM */ },
```

### Modus 4: code_review (`any`-Verwendung)

```bash
grep -rE ":\s*any\b|<any>" src/ --include="*.ts" | grep -v "// eslint-disable"
```

**Pass:** Keine ungerechtfertigten `any`-Verwendungen.
**Fail:** `any` für externe/LLM-Inputs.

## Pass Criteria

- [ ] `tsconfig.json` hat `"strict": true`
- [ ] Zod in `dependencies` (Version ≥ 3.0)
- [ ] **Alle** registrierten Tools verwenden `z.object(...)` als `inputSchema`
- [ ] Keine `any`-Typen für externe Eingaben (nur als ausnahme + ESLint-Disable mit Begründung)

## Common Failures

| Pattern | Risiko |
|---|---|
| Roh-JSON-Schema statt Zod | LLM-Args werden nicht validiert, Tool-Handler erhält Garbage |
| `any` für `args`-Parameter | Type-Sicherheit komplett umgangen |
| Zod-Schema, aber nicht verwendet (`.parse()` fehlt) | Validation nominell vorhanden, läuft aber nie |
| `tsconfig.json` mit `"strict": false` | Null-Safety fehlt, Optional Chaining wird oft vergessen |

## Remediation

```diff
+ import { z } from "zod";
+
+ const GetUserInfoArgs = z.object({
+   userId: z.string().uuid(),
+   includeArchived: z.boolean().optional(),
+ });
+
  server.registerTool({
    name: "getUserInfo",
    description: "Returns user profile data by ID.",
-   inputSchema: { type: "object", properties: { userId: { type: "string" } } },
+   inputSchema: GetUserInfoArgs,
    handler: async (args) => {
+     // args ist automatisch typed: { userId: string; includeArchived?: boolean }
      return await fetchUser(args.userId, args.includeArchived);
    },
  });
```

`tsconfig.json`:
```diff
  {
    "compilerOptions": {
-     "strict": false
+     "strict": true,
+     "noUncheckedIndexedAccess": true,
+     "exactOptionalPropertyTypes": true
    }
  }
```

## Effort

S–M — Bei kleinen Repos: 1 Tag. Bei grösseren mit vielen Tools: 2–3 Tage. Tests müssen angepasst werden, da `any`-Stellen plötzlich Type-Errors werfen.
