# CH-006 — Schulamt Klassifikationsschema

## Description

Die Stadt Zürich verwendet — analog zur Bundesverwaltung — ein vierstufiges Klassifikationsschema für Informationen:

| Stufe | Kürzel | Beispiele Schulamt |
|---|---|---|
| **Öffentlich** | — | Bildungsstatistiken auf data.stadt-zuerich.ch, Schulhaus-Adressen |
| **Intern (BUI)** | BUI | Stadtverwaltungs-interne Notizen, Lehrer:innen-Mailing-Listen |
| **Vertraulich** | VERT | Personalentscheide, Schullaufbahn-Daten von Schüler:innen |
| **Streng vertraulich** | SVERT | IF/IS-Klassifikation einzelner Schüler, gesundheitsbezogene Daten |

Ein MCP-Server muss explizit deklarieren, welche Stufen er verarbeitet. Die höchste Stufe bestimmt die Sicherheitsanforderungen — sowohl technisch (siehe CH-005 ISDS-Mapping) als auch organisatorisch (Zugangsbeschränkungen, Schulungs-Pflicht für Bearbeitende).

Bei `Schulamt-Kontext` ist dieser Check zwingend. Auch ein scheinbar harmloser Server kann durch Aggregation öffentlicher Daten implizit höhere Stufen erreichen — z.B. wenn die Kombination mehrerer Statistiken Rückschlüsse auf einzelne Klassen oder Schüler:innen erlaubt.

## Verification

### Modus 1: documentation_check (Klassifikations-Deklaration)

```bash
find . -iname '*klassifikation*' -o -iname '*classification*'
grep -iE 'BUI|vertraulich|streng vertraulich|öffentlich' README.md docs/
```

**Pass-Pattern (`docs/klassifikation.md`):**

```markdown
# Klassifikation: <server-name>

## Verarbeitete Daten-Klassen

| Datentyp | Stufe | Begründung |
|---|---|---|
| Schul-Statistiken aggregiert | Öffentlich | Bereits auf data.stadt-zuerich.ch publiziert |
| Klassen-Zuteilung (intern) | BUI | Nicht zur Veröffentlichung bestimmt |
| <weitere Datentypen> | <Stufe> | <Begründung> |

## Höchste Stufe: <BUI / VERT / SVERT>

## Aggregations-Risiko
<Falls relevant: Hinweis, ob Aggregation öffentlicher Daten zu höherer Stufe führt>

## Massnahmen-Mapping

### BUI
- TLS-Verschlüsselung in-transit
- Authentifizierung erforderlich
- Audit-Logs

### Vertraulich (zusätzlich)
- Verschlüsselung at-rest
- RBAC mit Need-to-Know-Prinzip
- Datenresidenz Schweiz oder explizit zugelassenes Drittland

### Streng vertraulich (zusätzlich)
- Hardware-Security-Module
- Air-Gap-Backup
- Vier-Augen-Prinzip bei administrativen Operationen
```

### Modus 2: code_review (Aggregations-Risiko explizit adressiert)

Wenn der Server öffentliche Daten aggregiert: prüfen, ob die Aggregations-Granularität explizit limitiert ist (z.B. minimale Klassen-/Schul-Grösse für Statistik-Outputs).

```bash
grep -rE 'k_anonymity|min_class_size|aggregation_threshold' src/
```

**Pass-Pattern (k-anonymity-light):**

```python
MIN_CLASS_SIZE_FOR_PUBLIC_STATS = 5

@mcp.tool()
async def class_statistics(school_id: str, ctx) -> dict:
    raw = await db.get_class_data(school_id)
    # Nur Aggregate für Klassen mit ≥ 5 Schüler:innen liefern
    aggregated = [
        c for c in raw if c["size"] >= MIN_CLASS_SIZE_FOR_PUBLIC_STATS
    ]
    return {
        "schools": aggregated,
        "filtered_classes_too_small": len(raw) - len(aggregated),
    }
```

## Pass Criteria

- [ ] Klassifikation pro Datentyp dokumentiert
- [ ] Höchste Stufe explizit deklariert
- [ ] Massnahmen-Mapping pro Stufe vorhanden
- [ ] Bei Aggregation: Granularitäts-Limit dokumentiert (k-anonymity oder Minimum-Grösse)
- [ ] Bei Stufe `Vertraulich` oder höher: Synergie zu CH-005 ISDS-Klassifikation hergestellt

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Pauschale Klassifikation «Öffentlich» trotz aggregierter Auswertungen | Re-Identifikation einzelner Schüler:innen möglich |
| Keine explizite Klassifikation | Operations-Team weiss nicht, wie Daten zu schützen sind |
| Klasse `BUI` aber Daten in Public-GitHub-Repo | Stufenverstoss |
| Aggregation kleiner Klassen ohne Schwelle | k-anonymity verletzt |

## Remediation

### Schritt 1: Inventar der Datentypen

Pro Tool des Servers durchgehen: was kommt rein, was geht raus, welche Stufe?

### Schritt 2: Klassifikations-Dokument erstellen

Wie im Pass-Pattern oben.

### Schritt 3: Aggregations-Limits implementieren

Bei aggregierenden Tools: minimale Gruppengrösse (Faustregel k = 5) als Hard-Filter.

### Schritt 4: Bei Stufe `Vertraulich` oder höher

Synergie zu CH-005: ISDS-Klassifikation muss «Vertraulichkeit Hoch» oder höher reflektieren. Genehmigung OIZ-Sicherheit nötig.

## Effort

S — < 1 Tag für Klassifikation. Bei Aggregations-Limit-Implementierung: zusätzlich 1 Tag.
