# SEC-007 — Container-Sandboxing

## Description

Lokale stdio-Server (siehe SEC-006) eliminieren die Netzwerk-Angriffsfläche, behalten aber das Risiko, dass ein kompromittierter Server-Code (durch Supply-Chain-Attack, böswilliges Update, oder Bug-Exploitation) mit User-Privilegien ausgeführt wird. Read-Zugriff auf `~/.ssh/`, `~/.aws/credentials`, Browser-Cookies, lokal gespeicherte Tokens — alles direkt erreichbar.

Defense-in-Depth: Server in einer Sandbox ausführen, die nur das nötige Minimum an Filesystem- und System-Call-Zugriff erlaubt:

| Mechanismus | Stärke | Aufwand | Wann |
|---|---|---|---|
| **Docker Container** | Mittel | Niedrig | Default für Cloud-Deployments |
| **Docker + seccomp + AppArmor** | Hoch | Mittel | Production mit sensitiven Tools |
| **gVisor / Firecracker** | Sehr hoch | Hoch | Multi-Tenant, untrusted Code |
| **chroot + UID-Drop** | Niedrig | Niedrig | Legacy-Pattern, nicht mehr empfohlen |
| **WASM Runtime** | Hoch | Hoch | Experimentell für MCP, 2026 noch Early |

Für Schulamt-Portfolio: Docker mit non-root User + read-only-Filesystem + seccomp-Default-Profile ist der pragmatische Standard.

## Verification

### Modus 1: code_review (Dockerfile-Hardening)

```bash
grep -E '^USER |^FROM .*scratch|^FROM .*distroless' Dockerfile
grep -E 'readOnlyRootFilesystem|allowPrivilegeEscalation|runAsNonRoot' k8s/ helm/
```

**Pass-Pattern (Dockerfile, baut auf SCALE-004 auf):**

```dockerfile
FROM python:3.11-slim AS builder
WORKDIR /build
COPY pyproject.toml .
RUN pip install --no-cache-dir --user -e .

FROM python:3.11-slim AS runtime
COPY --from=builder /root/.local /home/mcp/.local
COPY src/ /app/src/
WORKDIR /app

# Non-root user mit minimaler UID/GID
RUN useradd --uid 10001 --create-home --shell /usr/sbin/nologin mcp \
    && chown -R mcp:mcp /home/mcp /app

USER 10001
ENV PATH=/home/mcp/.local/bin:$PATH PYTHONUNBUFFERED=1

# Read-only-Filesystem-kompatibel: tmpfs für /tmp
VOLUME ["/tmp"]

HEALTHCHECK CMD curl -f http://localhost:8000/healthz || exit 1
CMD ["python", "-m", "server"]
```

**Pass-Pattern (Kubernetes SecurityContext):**

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 10001
        fsGroup: 10001
        seccompProfile:
          type: RuntimeDefault
      containers:
      - name: mcp-server
        image: malkreide/mcp-server:v0.1.0
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop: ["ALL"]
        volumeMounts:
        - name: tmp
          mountPath: /tmp
      volumes:
      - name: tmp
        emptyDir:
          sizeLimit: 100Mi
```

### Modus 2: runtime_test (Privileg-Reduktion verifizieren)

```bash
# Container-User
docker exec <container_id> id
# Erwartet: uid=10001(mcp) gid=10001(mcp), nicht uid=0(root)

# Filesystem read-only?
docker exec <container_id> sh -c "touch /etc/test 2>&1"
# Erwartet: "Read-only file system"

# Capabilities dropped?
docker exec <container_id> grep CapEff /proc/1/status
# Erwartet: CapEff = 0000000000000000
```

### Modus 3: code_review (Filesystem-Tools mit Allow-List)

Falls der Server File-Tools exponiert, muss der Pfad-Zugriff explizit limitiert sein (siehe SEC-017 für Path-Traversal-Schutz). Hier zusätzlich: das Container-Volume sollte nur den erlaubten Pfad mounten.

```yaml
volumeMounts:
- name: schulamt-data
  mountPath: /app/data
  readOnly: true  # falls nur Read-Tools
```

## Pass Criteria

- [ ] Dockerfile setzt `USER` mit nicht-root UID (≥ 10000)
- [ ] Bei Kubernetes: `runAsNonRoot: true`, `runAsUser` explizit, `allowPrivilegeEscalation: false`
- [ ] Bei Kubernetes: `readOnlyRootFilesystem: true` mit `tmpfs`-Volume für `/tmp`
- [ ] Bei Kubernetes: `capabilities.drop: ["ALL"]`
- [ ] seccomp-Profile mindestens `RuntimeDefault`
- [ ] Bei File-Tools: Volume-Mounts nur für erlaubte Pfade, ggf. read-only

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| `USER root` (Default) im Dockerfile | Container-Escape erleichtert |
| Kein `readOnlyRootFilesystem` | Malware kann Binary modifizieren |
| Capabilities nicht gedroppt | Linux-Capabilities erlauben Privileg-Eskalation |
| Volume mit `:rw` statt `:ro` für read-only-Daten | Unbeabsichtigte Schreib-Operationen |
| seccomp-Profil deaktiviert (`seccompProfile: Unconfined`) | Volle Syscall-Surface |

## Remediation

### Schritt 1: Dockerfile-User anpassen

Wie im Pass-Pattern oben.

### Schritt 2: Kubernetes-SecurityContext setzen

Im Helm-Chart oder Deployment-Manifest.

### Schritt 3: Tests gegen Privileg-Eskalation

```python
def test_container_runs_as_non_root():
    result = subprocess.run(
        ["docker", "exec", CONTAINER_ID, "id", "-u"],
        capture_output=True, text=True,
    )
    assert int(result.stdout.strip()) >= 10000

def test_filesystem_read_only():
    result = subprocess.run(
        ["docker", "exec", CONTAINER_ID, "touch", "/etc/test"],
        capture_output=True, text=True,
    )
    assert "Read-only" in result.stderr or result.returncode != 0
```

### Schritt 4: CI-Check via Trivy / Snyk

```yaml
- name: Container security scan
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: malkreide/mcp-server:${{ github.sha }}
    severity: CRITICAL,HIGH
    exit-code: 1
```
