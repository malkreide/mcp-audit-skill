# SEC-019 — Lethal Trifecta vermeiden

## Description

Simon Willisons «Lethal Trifecta»-Konzept beschreibt drei Fähigkeiten, die einzeln harmlos, **kombiniert** aber den Server zur Waffe in der Hand eines Prompt-Injection-Angreifers machen:

1. **Zugriff auf private Daten** (Verwaltungsdaten, PII, interne Dokumente)
2. **Exposition gegenüber untrusted Content** (User-Input, externe Dokumente, Web-Scraping)
3. **Externe Kommunikationsfähigkeit** (E-Mail-Versand, HTTP-POST, Slack, externe APIs)

Wenn alle drei in einem Server vereint sind, kann ein Angreifer via Prompt-Injection im untrusted Content den Server überzeugen, private Daten an externe Empfänger zu schicken. Klassisches Szenario: Server liest Mailbox des Users (private Daten), bekommt eine Phishing-Mail (untrusted Content), die ihn anweist alle Eingangsmails an `attacker@evil.com` weiterzuleiten — und kann das auch tun (externe Kommunikation).

Das Problem ist **strukturell ungelöst**. Es gibt keine technische Mitigation auf Code-Ebene — nur **Architektur-Disziplin**:

| Erlaubte Kombination | Beispiel |
|---|---|
| Read-only + untrusted Content | Reading-Server für Public-Data, kein Write |
| Read-only + externer Output (intern) | Read-Server, der nur an authentifizierte Clients antwortet |
| Write + Trusted Input only | Write-Server, dessen Input nur von menschlichen Admin-Calls kommt |

| Verbotene Kombination | Risiko |
|---|---|
| Read PII + lese externe URLs + sende Mails | Datenexfiltration via Prompt-Injection |
| Read DB + lese User-Uploads + HTTP-POST extern | DSG-Verstoss durch Prompt-Injection |
| Read Mailbox + sende Mails | Klassischer Mail-Worm-Vektor |

Die Lösung ist Server-Separation. Statt einen Mega-Server zu bauen, der alles kann: zwei oder drei kleinere Server, von denen jeder maximal **zwei** der drei Trifecta-Fähigkeiten hat. Der LLM-Host orchestriert die Zusammenarbeit, aber kein einzelner Server ist die Schwachstelle.

Im Schulamt-Kontext ist das aktuell wenig relevant (alle Server sind read-only), wird aber **kritisch**, sobald Write-Server dazukommen — etwa ein hypothetisches `notion-schulamt-mcp` mit Mailing-Funktion.

## Verification

### Modus 1: code_review (Trifecta-Mapping pro Server)

Für jeden Server erstellt man eine Trifecta-Bewertung:

```bash
# 1. Liest der Server private Daten?
grep -rE 'data_class|PII|Verwaltungsdaten' README.md docs/

# 2. Verarbeitet der Server untrusted Content?
grep -rE 'fetch|requests\.get|httpx\.|web_search|scrape|user.upload' src/

# 3. Sendet der Server an externe Empfänger?
grep -rE 'send_mail|smtplib|requests\.post|httpx\.post|slack|webhook' src/
```

**Pass-Pattern (Trifecta-Bewertung im README oder docs/):**

```markdown
## Lethal Trifecta Bewertung

| Fähigkeit | Status | Begründung |
|---|---|---|
| Zugriff auf private Daten | ❌ Nein | Nur Public Open Data von opendata.swiss |
| Exposition gegenüber untrusted Content | ⚠️ Eingeschränkt | Web-Scraping nur von whitelisted Domains |
| Externe Kommunikation | ❌ Nein | Read-only, kein Write/Send |

Trifecta-Score: 1 von 3 — sicher.
```

### Modus 2: documentation_check (Architecture Decision Record)

Bei Server-Splittung sollte eine Begründung dokumentiert sein:

```bash
find . -iname '*adr*' -o -iname '*architecture*' -o -iname '*trifecta*'
```

**Pass-Pattern (`docs/adr-001-server-separation.md`):**

```markdown
# ADR-001: Trennung Read- und Write-Server

## Status
Akzeptiert, 2026-04-26

## Kontext
Das ursprünglich geplante `schulamt-mcp` sollte sowohl Schul-Daten lesen
als auch Eltern-Mails versenden. Diese Kombination erfüllt zwei der drei
Lethal-Trifecta-Bedingungen (private Daten + externe Kommunikation).

## Entscheidung
Aufteilung in zwei Server:
- `schulamt-read-mcp`: read-only Zugriff auf Verwaltungsdaten
- `schulamt-mail-mcp`: Mail-Versand mit hartem Allow-List-Filter auf
  Empfänger-Domains (@schulen.zuerich.ch, @stadt-zuerich.ch)

Der LLM-Host orchestriert beide Server. Keiner alleine kann
Datenexfiltration durchführen.

## Konsequenzen
- Höhere Komplexität im Tool-Manifest (2 Server statt 1)
- Klare Trust-Boundaries
- Audit-Trail pro Server möglich
```

### Modus 3: code_review (Allow-Lists bei unvermeidbarer Kombination)

Falls die Kombination zweier Trifecta-Fähigkeiten unvermeidbar ist, muss die dritte hart eingeschränkt sein:

```bash
grep -rE 'allowed_recipients|allowed_domains|recipient_allowlist|domain_whitelist' src/
```

**Pass-Pattern:**

```python
ALLOWED_RECIPIENT_DOMAINS = frozenset({
    "schulen.zuerich.ch",
    "stadt-zuerich.ch",
    "schulamt.zuerich.ch",
})

@mcp.tool(annotations={"destructiveHint": False, "openWorldHint": True})
async def send_internal_notification(args: NotifyArgs, ctx: Context) -> dict:
    domain = args.recipient.split("@")[-1].lower()
    if domain not in ALLOWED_RECIPIENT_DOMAINS:
        return {"isError": True, "content": [TextContent(
            type="text",
            text=f"Recipient domain not in allow-list. "
                 f"Allowed: {sorted(ALLOWED_RECIPIENT_DOMAINS)}",
        )]}
    return await mailer.send(args.recipient, args.body)
```

## Pass Criteria

- [ ] Trifecta-Bewertung im README oder `docs/` dokumentiert
- [ ] Server hat maximal **zwei** der drei Trifecta-Fähigkeiten
- [ ] Bei zwei Fähigkeiten: ADR (Architecture Decision Record) dokumentiert die Begründung
- [ ] Bei drei Fähigkeiten (zu vermeiden): harte Allow-Lists für die dritte Fähigkeit, plus Sign-off durch Datenschutzbeauftragte:n
- [ ] Empfänger-Allow-Lists für Mail/Webhook-Sender als FrozenSet im Code, nicht config-mutierbar zur Laufzeit

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Mega-Server mit Read+Write+Send | Datenexfiltration via Prompt-Injection garantiert möglich |
| Allow-List konfigurierbar via ENV-Var | Operator-Fehler oder Kompromittierung kippt den Schutz |
| Allow-List als Blacklist («alles ausser X») | Neue Domains werden automatisch erlaubt |
| Kein Trifecta-Bewertungs-Dokument | Maintainer-Wechsel verliert das Wissen über Designentscheidung |

## Remediation

### Schritt 1: Trifecta-Audit pro Server

Für jeden Server im Portfolio die drei Fragen beantworten:

| Frage | Antwort | Score-Beitrag |
|---|---|---|
| Liest privater Daten? | ja/nein | +1 wenn ja |
| Untrusted Content? | ja/nein | +1 wenn ja |
| Externe Kommunikation? | ja/nein | +1 wenn ja |

Score 0–1: sicher. Score 2: ADR + Compensating Controls. Score 3: Server splitten.

### Schritt 2: Server-Splittung (bei Score 3)

Beispiel — aus einem hypothetischen `eltern-comm-mcp`:

```diff
- # Vorher: ein Server liest UND sendet
- @mcp.tool() def get_eltern_data(klassenid): ...
- @mcp.tool() def send_eltern_mail(recipient, body): ...

+ # Nachher: zwei Server
+ # eltern-data-mcp/
+ @mcp.tool() def get_eltern_data(klassenid): ...
+
+ # eltern-mail-mcp/  (separater Repo, separate Service-Identity)
+ ALLOWED_DOMAINS = frozenset({"schulen.zuerich.ch"})
+ @mcp.tool() def send_eltern_mail(recipient, body):
+     if recipient.split("@")[-1] not in ALLOWED_DOMAINS:
+         raise PermissionError(...)
```

### Schritt 3: ADR dokumentieren

Wie im Pass-Pattern Modus 2.

### Schritt 4: Audit-Trail

Bei Score-2-Servern: alle Tool-Calls werden geloggt, SIEM-Alerts (siehe OBS-005) auf ungewöhnliche Pattern (z.B. Recipients ausserhalb Allow-List).

## Effort

L — 1–2 Wochen bei nötiger Server-Splittung. S — < 1 Tag für reine Bewertung und ADR.
