# SEC-017 — Path-Traversal-Prevention

## Description

Wenn ein MCP-Server File-Tools exponiert (`read_file`, `list_directory`, `write_file`), wird der Pfad-Parameter aus LLM-generierten Args zum Sicherheits-Boundary. Empirisch sind Path-Traversal-Bugs der häufigste CVE-Typ in MCP-Server-Implementierungen 2025/2026 — mehrere offizielle Reference-Implementierungen waren betroffen.

Klassische Angriffs-Patterns:

| Variante | Beispiel | Wirkung |
|---|---|---|
| **Naiv** | `../../../etc/passwd` | Read ausserhalb erlaubtem Pfad |
| **URL-encoded** | `%2E%2E%2F%2E%2E%2Fetc%2Fpasswd` | Umgeht String-Match |
| **Doppel-encoded** | `%252E%252E` | Umgeht single-decode |
| **Unicode** | `..%c0%af..%c0%afetc/passwd` | Overlong UTF-8 |
| **Backslash (Windows)** | `..\..\windows\system32` | Auf Linux: Backslash literal, auf Windows: Traversal |
| **Symlink** | erlaubter Pfad zeigt via Symlink raus | Umgeht Pfad-Check |
| **Null-Byte** | `safe.txt\x00../../etc/passwd` | C-Style-Library bricht bei `\x00` ab |
| **Absolute Pfade** | `/etc/passwd` | Falls nicht expliziter Allow-Pattern |

Defense-Pattern: **Pfad-Resolution gegen Allow-Liste**, nicht String-Filtering. Konkret:

1. User-Input zu absolutem Pfad auflösen (`Path.resolve()`, folgt Symlinks)
2. Geprüft ob resolved Path innerhalb erlaubter Root-Verzeichnisse liegt
3. Falls nein: Rejection ohne Information-Disclosure (kein Hinweis warum)

## Verification

### Modus 1: code_review (Pfad-Resolution + Allow-List)

```bash
grep -rE 'resolve\(\)|os\.path\.realpath|path\.normalize' src/
grep -rE 'Path\(|os\.path\.join' src/
```

**Pass-Pattern (Python):**

```python
from pathlib import Path
from typing import Annotated
from pydantic import BaseModel, StringConstraints

ALLOWED_ROOTS = [
    Path("/app/data").resolve(),
    Path("/app/cache").resolve(),
]

class ReadFileArgs(BaseModel):
    model_config = {"strict": True, "extra": "forbid"}
    path: Annotated[str, StringConstraints(min_length=1, max_length=4096)]

def safe_resolve(user_path: str) -> Path:
    """Resolve user path, ensure it's within an allowed root."""
    # Step 1: Reject explicit traversal markers (defense-in-depth, nicht primary)
    if "\x00" in user_path or "\\" in user_path:
        raise ValueError("Invalid path characters")

    # Step 2: Resolve to absolute path (follows symlinks, normalizes ..)
    candidate = Path(user_path).resolve(strict=False)

    # Step 3: Check against allow-list with strict prefix-match
    for root in ALLOWED_ROOTS:
        try:
            candidate.relative_to(root)
            return candidate
        except ValueError:
            continue

    raise PermissionError("Path outside allowed roots")


@mcp.tool()
async def read_file(args: ReadFileArgs, ctx: Context) -> dict:
    try:
        resolved = safe_resolve(args.path)
    except (ValueError, PermissionError):
        # Generische Fehlermeldung — kein Information-Leak über Allow-Roots
        return {
            "isError": True,
            "content": [TextContent(
                type="text",
                text="File not accessible.",
            )],
        }
    return {"content": resolved.read_text(encoding="utf-8")}
```

**Fail-Pattern:**

```python
# FAIL: String-basiert, umgehbar
@mcp.tool()
async def read_file(path: str):
    if ".." in path:
        return {"error": "no traversal"}
    return open(path).read()  # /etc/passwd? aboslute path? all allowed
```

### Modus 2: code_review (Symlink-Verhalten)

`Path.resolve()` folgt Symlinks. Wenn der erlaubte Root einen Symlink ausserhalb der Root enthält, wird das aufgelöst — und der Pfad-Check schlägt korrekt fehl. Aber: bei `strict=True` wirft `resolve()` `FileNotFoundError`, was den Check umgehen könnte. Daher: erst resolve, dann existenz-prüfen.

### Modus 3: runtime_test (Angriffs-Patterns abgewiesen)

```python
@pytest.mark.parametrize("attack_path", [
    "../../../etc/passwd",
    "..%2F..%2Fetc%2Fpasswd",
    "/etc/passwd",
    "/app/data/../../../etc/passwd",
    "/app/data/legitimate.txt\x00../../etc/passwd",
    "/app/data/../../../../tmp/evil",
])
async def test_path_traversal_blocked(attack_path):
    args = ReadFileArgs(path=attack_path)
    result = await read_file(args, mock_ctx())
    assert result.get("isError") is True
```

### Modus 4: code_review (Write-Tools strenger als Read-Tools)

Write-Tools brauchen zusätzlich:

- Pfad-Existenz-Check (kein Overwrite ohne Plan)
- Atomic Write (temp-File + rename) statt direktem Write
- File-Size-Limits

## Pass Criteria

- [ ] Allow-List mit absoluten, resolved Roots definiert
- [ ] User-Pfad wird via `Path.resolve()` (oder Äquivalent) normalisiert
- [ ] Resolved Path wird gegen Allow-List geprüft (`relative_to` ohne Exception = im Allow-Bereich)
- [ ] Bei Mismatch: generische Fehlermeldung, kein Information-Leak
- [ ] Null-Byte und Backslash explizit gefiltert
- [ ] Tests decken alle Angriffs-Varianten ab (parametrized)
- [ ] Bei Write-Tools: atomic-rename + Size-Limits

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| `if ".." in path: reject` | Umgehbar via Encoding |
| `os.path.join(root, user_input)` ohne Resolve-Check | Absolute User-Pfad überschreibt Root |
| `os.path.normpath()` allein ohne Allow-List-Check | Normalisiert, aber prüft nicht |
| Fehlermeldung nennt Allow-Roots | Information-Leak |
| Symlink-Resolution disabled | Symlinks zeigen aus Allow-List heraus |
| Write-Tool ohne Atomic-Rename | Partial-Write bei Crash → korrupte Files |

## Remediation

### Schritt 1: Allow-List definieren

```python
# Bei Setup
ALLOWED_ROOTS: list[Path] = [
    Path(os.environ["DATA_ROOT"]).resolve(),
    Path(os.environ["CACHE_ROOT"]).resolve(),
]
# Logging beim Start zur Verifikation
logger.info("filesystem_allowlist", roots=[str(r) for r in ALLOWED_ROOTS])
```

### Schritt 2: `safe_resolve()` als Helper

```python
def safe_resolve(user_path: str, write: bool = False) -> Path:
    if "\x00" in user_path or "\\" in user_path:
        raise ValueError("invalid_chars")
    candidate = Path(user_path).resolve(strict=False)
    for root in ALLOWED_ROOTS:
        try:
            candidate.relative_to(root)
            if write and not _is_writable_root(root):
                raise PermissionError("read_only_root")
            return candidate
        except ValueError:
            continue
    raise PermissionError("outside_allowlist")
```

### Schritt 3: Atomic Write bei Schreib-Tools

```python
import tempfile
import os

def atomic_write(path: Path, content: bytes):
    fd, tmp_path = tempfile.mkstemp(dir=path.parent, prefix=f".{path.name}.")
    try:
        with os.fdopen(fd, "wb") as f:
            f.write(content)
        os.replace(tmp_path, path)  # atomic on POSIX
    except Exception:
        os.unlink(tmp_path)
        raise
```
