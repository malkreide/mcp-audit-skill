# SEC-010 — OAuth State Parameter: Single-Use, kurze TTL

## Description

Der OAuth-`state`-Parameter dient drei Zwecken:

1. **CSRF-Schutz:** Verbindet eine Authorization-Anfrage mit dem späteren Callback. Ohne `state` kann ein Angreifer einen Authorization-Code an die Callback-URL des Opfers schicken und damit dessen Session übernehmen.
2. **Session-Resumption:** Ermöglicht dem Server, den Kontext (z.B. ursprünglich angeforderter Scope, Client-ID) beim Callback wiederherzustellen.
3. **Replay-Schutz:** Verhindert, dass ein einmal gültiger Authorization-Code mehrfach eingelöst wird.

Punkt 3 ist nur erfüllt, wenn der `state`-Parameter **single-use** ist und nach erstem erfolgreichem Callback serverseitig gelöscht wird. Sonst kann ein Angreifer, der einmal an `code` + `state` kommt (z.B. durch Log-Leak, Browser-History, Referer-Header), diese Kombination wiederverwenden.

Zusätzlich: Die TTL muss kurz sein (max 10 Minuten). Längere Lebensdauer erhöht das Replay-Fenster ohne legitimen Nutzen — User klicken Consent-Buttons innerhalb von Sekunden, nicht Stunden.

## Verification

### Modus 1: code_review (Generierung kryptografisch sicher)

```bash
grep -rE "state.*token|oauth.*state|generate.*state" src/
grep -rE "secrets\.|crypto\.|random_urlsafe|token_urlsafe" src/
```

**Pass-Pattern:**

```python
import secrets

def generate_oauth_state() -> str:
    return secrets.token_urlsafe(32)  # 256 bit entropy
```

**Fail-Pattern:**

```python
# FAIL: Math.random / vorhersagbar
state = str(uuid.uuid1())  # ← v1 enthält Timestamp + MAC, vorhersagbar
state = str(int(time.time())) + "_" + str(random.randint(1, 1000))
```

### Modus 2: code_review (Single-Use durch Server-Side-Storage)

```bash
# Suche nach State-Storage-Logik
grep -rE "state.*store|store.*state|state.*cache|state.*redis" src/
```

**Pass-Pattern:**

```python
import secrets
from datetime import timedelta
from redis.asyncio import Redis

class OAuthStateStore:
    def __init__(self, redis: Redis, ttl: timedelta = timedelta(minutes=10)):
        self.redis = redis
        self.ttl = ttl

    async def issue(self, client_id: str, redirect_uri: str, scopes: list[str]) -> str:
        state = secrets.token_urlsafe(32)
        await self.redis.setex(
            name=f"oauth:state:{state}",
            time=int(self.ttl.total_seconds()),
            value=json.dumps({
                "client_id": client_id,
                "redirect_uri": redirect_uri,
                "scopes": scopes,
                "issued_at": time.time(),
            }),
        )
        return state

    async def consume(self, state: str) -> dict | None:
        """Atomically read AND delete. Returns None if state was already used or expired."""
        # GETDEL: atomic read-and-delete (Redis 6.2+)
        raw = await self.redis.execute_command("GETDEL", f"oauth:state:{state}")
        if raw is None:
            return None
        return json.loads(raw)
```

**Verwendung im Callback-Handler:**

```python
@app.get("/oauth/callback")
async def callback(code: str, state: str):
    state_data = await state_store.consume(state)
    if state_data is None:
        # State unbekannt, abgelaufen, oder bereits verbraucht
        raise HTTPException(401, "Invalid or expired state")
    # state_data ist jetzt aus Redis verschwunden, Replay nicht möglich
    return await exchange_code_for_token(code, state_data)
```

**Fail-Pattern:**

```python
# FAIL: Get ohne Delete → Replay möglich
state_data = await redis.get(f"oauth:state:{state}")
if state_data:
    return await exchange_code_for_token(code, json.loads(state_data))
# State bleibt in Redis — Angreifer kann mit gleichem state+code nochmal aufrufen

# FAIL: Get + Delete als zwei separate Operations → Race Condition
state_data = await redis.get(f"oauth:state:{state}")
await redis.delete(f"oauth:state:{state}")  # zwei parallele Callbacks könnten beide GET vor DELETE machen
```

### Modus 3: code_review (TTL ≤ 10 Minuten)

```bash
grep -rE "ttl|expire|max_age" src/ | grep -i "state\|oauth"
```

**Pass:** TTL-Wert ist im Code sichtbar und ≤ 600 Sekunden.

**Fail:** TTL ist 1h, 1d, oder gar nicht gesetzt.

### Modus 4: runtime_test (Replay-Versuch wird abgelehnt)

```bash
# Schritt 1: Legitimer Flow
STATE=$(curl -s "https://server/oauth/authorize?client_id=test&scope=read" \
  | grep -oP 'state=\K[^&]+')
# (User klickt Consent, wird zu callback redirected)
curl -s "https://server/oauth/callback?code=ABC&state=$STATE"
# → HTTP 200, Token erhalten

# Schritt 2: Replay
curl -s "https://server/oauth/callback?code=ABC&state=$STATE" \
  -w "\nHTTP %{http_code}\n"
# Erwartet: HTTP 401 — state bereits verbraucht
```

## Pass Criteria

- [ ] State wird mit `secrets.token_urlsafe(32)` oder gleichwertig generiert (≥256 bit Entropie)
- [ ] State wird **vor** dem Authorization-Request serverseitig gespeichert (nicht client-side)
- [ ] Callback-Handler verwendet **atomic read-and-delete** (Redis `GETDEL` oder DB-Transaktion)
- [ ] TTL ≤ 600 Sekunden (10 Minuten)
- [ ] Bei Mismatch / Expiry / Replay: HTTP 401 mit klarer Fehlermeldung
- [ ] State-Storage ist isoliert pro Authorization-Server-Instanz (kein Multi-Tenant-Leak)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| State im Cookie statt serverseitig | Client kann State manipulieren |
| `GET` + `DELETE` als zwei Operations | Race Condition: paralleler Replay möglich |
| TTL > 10min | Vergrössertes Replay-Fenster |
| State wird bei Authorization-Request **nicht** persistiert | Callback kann keine Validation durchführen |
| State + Code in URL-Logs sichtbar | Nginx/CloudFront Access-Logs als Leak-Vector |

## Remediation

Volles Pattern oben (Modus 2). Zusätzlich:

### State-Erzeugung absichern gegen Brute-Force

Falls aus Compliance-Gründen kürzere States nötig (z.B. mobile-Friendly), zumindest mit Server-Side Rate-Limiting absichern:

```python
@app.get("/oauth/callback")
@rate_limit("10/minute")  # max 10 Callback-Versuche pro IP/min
async def callback(code: str, state: str):
    ...
```

### State + Code aus Logs entfernen

```nginx
# nginx
log_format minimal '$remote_addr - $time_local "$request_method $uri" '
                   '$status $body_bytes_sent';
# Statt $request — vermeidet Logging der Query-Strings inkl. state/code

access_log /var/log/nginx/access.log minimal;
```

### CI-Test gegen Replay

```python
@pytest.mark.asyncio
async def test_oauth_state_is_single_use(client, redis):
    state = await state_store.issue("client_a", "https://...", ["read"])
    response1 = await client.get(f"/oauth/callback?code=valid&state={state}")
    assert response1.status_code == 200

    response2 = await client.get(f"/oauth/callback?code=valid&state={state}")
    assert response2.status_code == 401
    assert "state" in response2.json()["error"].lower()
```
