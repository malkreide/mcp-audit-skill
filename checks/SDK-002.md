# SDK-002 — Strukturierte Tool-Returns

## Description

FastMCP wraps Tool-Returns automatisch in MCP-konformes Format — aber nur, wenn der Return-Typ strukturiert ist. Bei plain `dict` oder `str` muss FastMCP raten, welche Felder optional sind, welche Validierungen gelten, was passiert wenn Schema-Mismatches auftreten. Bei Pydantic-`BaseModel`, `TypedDict` oder `@dataclass` ist alles explizit und typgeprüft.

Konkrete Vorteile:

1. **Automatische Schema-Generierung:** FastMCP exponiert das Output-Schema im `tools/list`-Manifest. Das LLM weiss damit, was es erwarten kann, und kann Folge-Calls präziser planen.
2. **Runtime-Validation:** Wenn der Tool-Handler versehentlich ein Feld vergisst oder einen falschen Typ liefert, schlägt das früh fehl — nicht erst beim LLM-Parsing.
3. **Provenance-Envelope (siehe `mcp-data-source-probe`):** Konsistenter Pattern für `source` + `provenance` über alle Tools.

## Verification

### Modus 1: code_review (Return-Type-Annotations)

```bash
# Suche nach Tool-Definitionen und prüfe Return-Annotations
grep -rB2 -A20 "@mcp\.tool" src/ | grep -E "-> |async def.*->"
```

**Pass-Pattern (Pydantic):**

```python
from pydantic import BaseModel, Field
from typing import Literal

class MotionResult(BaseModel):
    """Pro parlamentarischer Motion."""
    id: str
    title: str
    eingebracht_von: str
    datum: str
    status: Literal["eingereicht", "behandelt", "abgelehnt", "angenommen"]
    tags: list[str] = Field(default_factory=list)

class SearchResponse(BaseModel):
    """Standard-Envelope für Such-Tools."""
    source: str = Field(default="Curia Vista — CC BY 4.0")
    provenance: Literal["live_api", "cached", "weekly_dump"]
    match_type: Literal["exact", "fuzzy", "none"]
    results: list[MotionResult]
    count: int

@mcp.tool()
async def search_motions(query: str, ctx) -> SearchResponse:
    raw = await api.search(query)
    return SearchResponse(
        provenance="live_api",
        match_type="exact" if raw else "none",
        results=[MotionResult(**r) for r in raw],
        count=len(raw),
    )
```

**Pass-Pattern (TypedDict, leichter):**

```python
from typing import TypedDict, Literal

class SearchResponse(TypedDict):
    source: str
    provenance: Literal["live_api", "cached"]
    results: list[dict]
    count: int

@mcp.tool()
async def search(query: str) -> SearchResponse:
    return {
        "source": "Curia Vista",
        "provenance": "live_api",
        "results": [...],
        "count": 0,
    }
```

**Fail-Pattern:**

```python
@mcp.tool()
async def search(query: str):  # ← keine Return-Annotation
    return {"results": [...]}  # plain dict, FastMCP rät Schema
```

### Modus 2: automated (Pydantic-Version)

```bash
grep -E "pydantic\s*[>=~]?=\s*[\"']?[12]" pyproject.toml requirements.txt 2>/dev/null
python -c "import pydantic; print(pydantic.VERSION)"
```

**Pass:** Pydantic ≥ 2.0. Pydantic v1 hat andere API (`.parse_obj`, `BaseModel.Config`-Klasse), die nicht mehr empfohlen ist.

## Pass Criteria

- [ ] Pydantic ≥ 2.0 in `dependencies`
- [ ] Tools haben **explizite Return-Annotations** (BaseModel / TypedDict / Dataclass / `dict[str, X]`)
- [ ] Such-/Listen-Tools verwenden einen konsistenten Response-Envelope mit `source`, `provenance`, `results`, `count`
- [ ] Bei Pydantic: Felder mit `Field(default=...)` oder `Field(default_factory=...)` für Defaults
- [ ] Literal-Types für enumerable-Werte (status, match_type, etc.) statt `str`

## Common Failures

| Pattern | Risiko |
|---|---|
| Plain `dict` als Return ohne Annotation | FastMCP exponiert kein Schema, LLM rät |
| Pydantic v1-Syntax (`.parse_obj`, `Config`-Klasse) | Deprecated, Migration nötig |
| Konsistenz fehlt: manche Tools mit Envelope, andere ohne | LLM kann Pattern nicht lernen |
| Mutable Defaults (`= []`) statt `Field(default_factory=list)` | Klassischer Python-Bug, geteilter State |

## Remediation

```diff
+ from pydantic import BaseModel, Field
+ from typing import Literal
+
+ class SearchResponse(BaseModel):
+     source: str = Field(default="DataSource Name — CC BY 4.0")
+     provenance: Literal["live_api", "cached", "weekly_dump"]
+     results: list[dict]
+     count: int

  @mcp.tool()
- async def search(query: str):
-     results = await api.search(query)
-     return {"results": results, "count": len(results)}
+ async def search(query: str, ctx) -> SearchResponse:
+     results = await api.search(query)
+     return SearchResponse(
+         provenance="live_api",
+         results=results,
+         count=len(results),
+     )
```
