# OPS-001 — Test-Strategie

## Description

Aus dem Sormena-Pattern bewährt: zwei Test-Kategorien mit klarer Trennung.

| Kategorie | Zweck | Wann ausgeführt | Mock | Speed |
|---|---|---|---|---|
| **Unit-Tests** | Server-Logik isoliert prüfen | CI bei jedem PR | respx-mocked HTTP | ~1s pro Test |
| **Live-Tests** | Echte API-Antworten gegen aktuelle Schnittstellen prüfen | Manuell, nightly, vor Release | keiner | 5-30s pro Test |

Die Trennung ist nicht akademisch — sie löst drei reale Probleme:

1. **CI-Stabilität:** Live-Tests scheitern bei API-Outages der Datenquelle (z.B. opendata.swiss-Wartung). Wenn alle Tests in CI laufen, wird der Server für stable-merge unbrauchbar.
2. **API-Quote-Schutz:** Manche Datenquellen haben Rate-Limits. Live-Tests bei jedem PR brennen Quote, die für Production-Traffic nötig wäre.
3. **Schema-Drift-Detection:** Live-Tests fangen Änderungen der Quell-API auf, die unit-Tests nicht sehen würden — aber sie brauchen Zeit, idealerweise nightly.

Implementation in Python mit pytest:

- Unit-Tests: `tests/test_unit.py` mit `respx` für HTTP-Mocking
- Live-Tests: `tests/test_live.py` mit `@pytest.mark.live`-Decorator
- CI-Workflow: `pytest -m "not live"` (skipt live-Tests)
- Manuell: `pytest -m live` oder `pytest tests/test_live.py`

Dieser Check ist `high`, weil ohne saubere Test-Strategie:

- Refactorings unsicher sind (keine Regression-Detection)
- API-Schema-Drift unbemerkt bleibt
- CI-Pipeline unzuverlässig wird (random Live-Failures)

## Verification

### Modus 1: code_review (Test-Verzeichnis-Layout)

```bash
ls tests/ 2>/dev/null
test -f tests/test_unit.py && echo "✓ unit" || echo "✗ unit MISSING"
test -f tests/test_live.py && echo "✓ live" || echo "✗ live MISSING"
```

**Pass:** Beide Files vorhanden. Mindestens 5 Unit-Tests und 1 Live-Test pro Tool.

### Modus 2: code_review (respx-Mocking in Unit-Tests)

```bash
grep -rE 'import respx|@respx\.|@pytest\.mark\.respx' tests/
```

**Pass-Pattern:**

```python
# tests/test_unit.py
import pytest
import respx
import httpx
from server_name.tools.search import search_motions


@respx.mock
async def test_search_returns_normalized_results():
    respx.get("https://ws.curia-vista.parlament.ch/odata/Motion") \
        .respond(200, json={
            "value": [
                {"Id": 123, "TitleDe": "Test", "Status": "submitted"},
            ],
        })
    result = await search_motions(query="test")
    assert len(result) == 1
    assert result[0]["title"] == "Test"


@respx.mock
async def test_search_handles_503():
    respx.get("https://ws.curia-vista.parlament.ch/odata/Motion") \
        .respond(503, text="Service Unavailable")
    result = await search_motions(query="test")
    assert result.get("isError")
```

### Modus 3: code_review (Live-Test-Marker)

```bash
grep -rE '@pytest\.mark\.live|pytest\.mark\.live' tests/
grep -rE 'markers' pyproject.toml setup.cfg pytest.ini
```

**Pass-Pattern:**

```python
# tests/test_live.py
import pytest
from server_name.tools.search import search_motions


@pytest.mark.live
async def test_real_curia_vista_search():
    result = await search_motions(query="Bildung")
    assert isinstance(result, list)
    assert len(result) > 0
    # Schema-Sanity: Live-API muss erwartete Felder liefern
    assert "title" in result[0]
    assert "status" in result[0]
```

**`pyproject.toml` muss den Marker registrieren:**

```toml
[tool.pytest.ini_options]
markers = [
    "live: tests against real APIs (manual, nightly only)",
]
```

### Modus 4: config_check (CI excludiert Live-Tests)

```bash
grep -rE 'pytest.*-m\s+"not live"|pytest.*--ignore.*live' .github/workflows/
```

**Pass-Pattern (`.github/workflows/test.yml`):**

```yaml
- name: Run unit tests (no live)
  run: pytest -m "not live" --cov=src --cov-report=term-missing
```

Plus separater Live-Workflow nightly:

```yaml
# .github/workflows/live-test.yml
on:
  schedule:
    - cron: "0 4 * * *"  # nightly 04:00 UTC
  workflow_dispatch:

jobs:
  live-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-python@v6
      - run: pip install -e ".[dev]"
      - run: pytest -m live
        env:
          # Test-spezifische Credentials, nicht die Production-API-Keys
          UPSTREAM_API_KEY: ${{ secrets.LIVE_TEST_API_KEY }}
```

## Pass Criteria

- [ ] `tests/test_unit.py` mit mindestens 5 Unit-Tests pro Tool
- [ ] `tests/test_live.py` mit mindestens 1 Live-Test pro Tool
- [ ] Unit-Tests verwenden `respx` (Python) oder `nock` (Node) für HTTP-Mocking
- [ ] Live-Tests sind mit `@pytest.mark.live` (oder Äquivalent) markiert
- [ ] Marker im `pyproject.toml` registriert
- [ ] CI-Workflow läuft `pytest -m "not live"`
- [ ] Separater nightly/manueller Live-Test-Workflow
- [ ] Live-Tests nutzen Test-spezifische Credentials, nicht Production-Keys (Synergie zu SEC-013)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Keine Tests | Regressionen unbemerkt, Refactoring riskant |
| Tests rufen Live-API in CI | CI bricht bei API-Outages |
| Kein respx-Mock — Tests nicht reproduzierbar | Flakiness |
| Live-Tests ohne Marker | Können nicht selektiv ausgeschlossen werden |
| Live-Tests mit Production-API-Keys | Quote-Verbrauch, Audit-Verschmutzung |

## Remediation

### Schritt 1: pyproject.toml-Marker registrieren

```toml
[tool.pytest.ini_options]
markers = [
    "live: tests against real APIs (manual, nightly only)",
]
```

### Schritt 2: respx als Dev-Dependency

```toml
[project.optional-dependencies]
dev = [
    "pytest >= 7.4",
    "pytest-asyncio >= 0.21",
    "pytest-cov >= 4.1",
    "respx >= 0.20",
]
```

### Schritt 3: Unit-Test-Suite aufbauen

Pro Tool mindestens drei Tests:
- Happy-Path (200, expected schema)
- Error-Path (4xx/5xx)
- Edge-Case (leere Antwort, malformed input)

### Schritt 4: CI-Workflow updaten

`.github/workflows/test.yml`:

```yaml
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-python@v6
        with:
          python-version: "3.11"
      - run: pip install -e ".[dev]"
      - run: pytest -m "not live" --cov=src
```

### Schritt 5: Nightly-Live-Workflow

Wie im Pass-Pattern Modus 4.

## Effort

M — 1–3 Tage Initial-Setup. Tests-Schreiben skaliert mit Tool-Anzahl.
