# SEC-003 — Progressive Scope-Minimierung

## Description

OAuth-Scope-Design hat zwei Failure-Modes:

1. **Zu permissiv (Omnibus-Scopes):** `files:*`, `db:*`, `admin` werden bei der ersten Authorization angefragt. Wenn der Token kompromittiert wird, hat der Angreifer Vollzugriff. Auch ohne Kompromittierung verletzt der Server das Least-Privilege-Prinzip.

2. **Zu restriktiv (Consent Fatigue):** Bei jeder neuen Operation wird der User mit einem Re-Auth-Dialog konfrontiert. User klicken «OK» blind oder brechen die Interaktion ab.

Best Practice (PDF Sec 4.3): **Progressive Scope-Anforderung**. Der Server startet mit einem minimalen Discovery-Scope (`mcp:tools-basic`). Wenn ein User eine privilegierte Operation aufruft, sendet der Server einen `WWW-Authenticate`-Header mit dem benötigten Scope zurück. Der Client führt einen inkrementellen Auth-Flow durch, der nur diesen einen Scope hinzufügt — ohne den User abzumelden.

Implementierung erfordert:

1. **Scope-Hierarchie definieren** — granulare Scopes statt Omnibus
2. **Server-seitige Scope-Validation** pro Tool-Call
3. **WWW-Authenticate-Challenge** mit konkretem Scope bei Insufficient-Scope-Error
4. **Client-Cache abgelehnter Scopes** — User soll nicht 5x dasselbe abgelehnte Scope sehen

### Modus 1: code_review (Scope-Hierarchie definiert)

```bash
grep -rE 'scope|Scope|OAuthScope' src/ | grep -v -E '\.test|tests/'
```

**Pass-Pattern (Scope-Definitionen):**

```python
from enum import StrEnum

class Scope(StrEnum):
    # Discovery — minimal, beim ersten Login
    TOOLS_BASIC = "mcp:tools-basic"

    # Read-Scopes — granular pro Domäne
    DATA_READ_PUBLIC = "data:read:public"
    DATA_READ_INTERNAL = "data:read:internal"
    DATA_READ_PII = "data:read:pii"

    # Write-Scopes — separate Hierarchie
    DATA_WRITE_PUBLIC = "data:write:public"
    DATA_WRITE_INTERNAL = "data:write:internal"

    # Admin — explizit, nicht in Omnibus
    ADMIN_USER_MGMT = "admin:user-management"


# Pro Tool: required scopes
TOOL_SCOPES = {
    "search": [Scope.DATA_READ_PUBLIC],
    "search_pii": [Scope.DATA_READ_PII],
    "update_config": [Scope.DATA_WRITE_INTERNAL],
    "delete_user": [Scope.ADMIN_USER_MGMT],
}
```

**Fail-Pattern:**

```python
# FAIL: Omnibus-Scope für alles
REQUIRED_SCOPE = "schulamt:full"  # zu breit
```

### Modus 2: code_review (WWW-Authenticate-Challenge)

```bash
grep -rE 'WWW-Authenticate|insufficient_scope|401.*scope' src/
```

**Pass-Pattern:**

```python
from fastapi import HTTPException
from fastapi.responses import JSONResponse

class InsufficientScopeError(HTTPException):
    def __init__(self, required_scope: str):
        super().__init__(
            status_code=403,
            detail={"error": "insufficient_scope", "scope": required_scope},
            headers={
                "WWW-Authenticate": (
                    f'Bearer realm="schulamt-mcp", '
                    f'error="insufficient_scope", '
                    f'scope="{required_scope}"'
                )
            },
        )


def require_scopes(*required: Scope):
    """Decorator-Factory für Scope-Validation pro Tool."""
    def decorator(tool_func):
        @functools.wraps(tool_func)
        async def wrapper(args, ctx: Context):
            claims = ctx.user_claims  # aus SEC-002
            user_scopes = set(claims.get("scope", "").split())
            missing = [s for s in required if s not in user_scopes]
            if missing:
                raise InsufficientScopeError(required_scope=missing[0])
            return await tool_func(args, ctx)
        return wrapper
    return decorator


@mcp.tool()
@require_scopes(Scope.DATA_READ_PII)
async def search_pii(args, ctx):
    return await db.search_pii(args.query)
```

```markdown
## OAuth Scopes

| Scope | Bedeutung | Anfrage-Zeitpunkt |
|---|---|---|
| `mcp:tools-basic` | Tool-Discovery, Read-Only-Search auf Public Data | Initial Login |
| `data:read:public` | Lesen von Public-Open-Data | Auf Anforderung |
| `data:read:internal` | Lesen von Verwaltungsdaten | Inkrementell, mit Begründung |
| `data:read:pii` | Lesen von Personendaten | Streng begrenzt, nur für autorisierte Rollen |
| `admin:*` | Administrative Operationen | Separate Authorization, nicht im Standard-Flow |

Der Server fordert Scopes inkrementell an. Bei einer Operation, für die du
keinen Scope hast, siehst du einen Re-Auth-Dialog mit dem konkreten Scope —
nicht mit allen Scopes auf einmal.
```

## Pass Criteria

- [ ] Scope-Hierarchie ist definiert (kein Omnibus `*:*`)
- [ ] Pro Tool dokumentierte erforderliche Scopes
- [ ] Server validiert Scopes pro Tool-Call (nicht nur beim Login)
- [ ] Bei Insufficient-Scope: HTTP 403 mit `WWW-Authenticate`-Header und konkretem Scope
- [ ] Initial-Login-Scope ist minimal (Discovery + Public-Read)
- [ ] Granularität: lese/schreibe + Datenklasse als separate Dimensionen
- [ ] Admin-Scopes sind explizit, nicht in Standard-Hierarchie eingebettet

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Omnibus-Scope `app:full` | Token-Leak = Total-Compromise |
| Initial-Login fragt alle möglichen Scopes ab | Consent Fatigue, User klickt durch |
| Scope-Validation nur am API-Gateway, nicht in Tool-Handler | Defense-in-Depth fehlt |
| Kein `WWW-Authenticate` bei Insufficient-Scope | Client kann nicht inkrementell auth-en |
| `admin:read` statt separate `admin:user-management`, `admin:audit-logs` | Zu grob, wieder Omnibus-Risiko |

## Remediation

### Schritt 1: Scope-Audit

Für alle Tools:

| Tool | Datentyp | Read/Write | Erforderlicher Scope |
|---|---|---|---|
| `search_education_stats` | Public | Read | `data:read:public` |
| `search_klassen` | Internal | Read | `data:read:internal` |
| `update_user_email` | Internal | Write | `data:write:internal` |
| `delete_user` | Admin | Write | `admin:user-management` |

### Schritt 2: Scope-Decorator implementieren

Wie im Pass-Pattern.

### Schritt 3: OAuth-Provider konfigurieren

Beim OAuth-Provider (SwitchAAI für Stadt Zürich, oder selbst-gehostete) die Scope-Liste registrieren.

### Schritt 4: Tests

```python
@pytest.mark.parametrize("user_scopes,tool,expected_status", [
    ("data:read:public", "search_education_stats", 200),
    ("data:read:public", "search_klassen", 403),
    ("data:read:internal", "search_klassen", 200),
    ("data:read:internal data:read:public", "search_klassen", 200),
])
async def test_scope_enforcement(client, user_scopes, tool, expected_status):
    token = create_test_jwt(scope=user_scopes)
    response = await client.post(
        f"/mcp/tools/{tool}",
        headers={"Authorization": f"Bearer {token}"},
    )
    assert response.status_code == expected_status
    if expected_status == 403:
        assert "WWW-Authenticate" in response.headers
```
