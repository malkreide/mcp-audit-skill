# OBS-005 — SIEM-Integration für Audit-Logs

## Description

MCP-Server, die Verwaltungsdaten oder PII verarbeiten, müssen einen revisionssicheren Audit-Trail führen. Operational Logs (Performance, Routine-Errors) und Audit Logs (Auth-Versuche, Token-Issuance, Scope-Erweiterungen, Schreib-Operationen, Tool-Registrierungs-Änderungen) sind unterschiedlich klassifiziert und werden idealerweise getrennt aggregiert.

Für Stadt Zürich Schulamt-Kontext: Audit-Logs gehen ins zentrale SIEM (vermutlich Splunk, je nach OIZ-Setup). Bei Cloud-Deployments ohne direkten SIEM-Zugang: Datadog EU-Region (siehe CH-001) als Zwischenlösung mit definierter Aufbewahrung und Export-Fähigkeit.

Dieser Check wird `high` oder `critical`, sobald PII verarbeitet wird (Schwelle: `data_class == "PII"`).

## Verification

### Modus 1: code_review (Audit-Logger getrennt von operational)

```bash
grep -rE 'audit_logger|AuditLog|audit\.' src/
grep -rE 'datadog|splunk|loki|honeycomb' src/ pyproject.toml package.json
```

**Pass-Pattern:**

```python
import structlog

operational_logger = structlog.get_logger("mcp.ops")
audit_logger = structlog.get_logger("mcp.audit")

@mcp.tool()
async def update_user_email(user_id: str, new_email: str, ctx) -> dict:
    # Audit-Log VOR der Operation (siehe HITL-005)
    audit_logger.warning(
        "user_email_change_initiated",
        target_user=user_id,
        new_email_domain=new_email.split("@")[1],  # Domain ok, Vollwert nicht
        actor_session=ctx.session_id,
        actor_client=ctx.client_info.name,
        correlation_id=ctx.request_id,
    )
    result = await api.update_email(user_id, new_email)
    audit_logger.warning(
        "user_email_change_completed",
        target_user=user_id,
        success=result.success,
        correlation_id=ctx.request_id,
    )
    return result
```

### Modus 2: documentation_check (Aufbewahrungs-Policy)

```bash
find . -iname 'audit-retention*' -o -iname 'log-retention*' -o -iname 'siem*'
```

**Pass:** Dokument vorhanden mit Aufbewahrungsfrist (≥ 1 Jahr für Verwaltungsdaten), Export-Verfahren, Zugriffs-Berechtigungen.

## Pass Criteria

- [ ] Audit-Logs sind klar von operational getrennt (separater Logger/Stream)
- [ ] Audit-Events enthalten: actor (user/session), action, target, timestamp, correlation_id
- [ ] Schreib-/destruktive Operationen werden VOR und NACH Ausführung geloggt
- [ ] SIEM-Export aktiv (Datadog EU / Splunk / OIZ-internes SIEM)
- [ ] Bei `Verwaltungsdaten`-Klasse: Aufbewahrung dokumentiert (idealerweise ≥ 1 Jahr für Compliance)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Audit + Operational im selben Log-Stream | Forensik-Suche unzuverlässig |
| PII in Audit-Logs (volle E-Mails, Klartext-Namen) | Audit selbst wird zur Datenschutz-Schwachstelle |
| Nur lokale Logs (Container-Filesystem) | Verlust bei Container-Restart |
| Audit nur NACH der Operation | Bei Crash mid-operation: kein Spur |
| SIEM in USA-Region | DSG-Verstoss (siehe CH-001) |

## Remediation

Minimal-Setup für Schulamt-Server:

1. Separater `audit_logger` (structlog get_logger mit eigenem Namespace)
2. Audit-Events in JSON-Format mit Pflichtfeldern: actor, action, target, timestamp, correlation_id, success
3. Per Datadog Agent (oder Splunk Forwarder) Export
4. Aufbewahrungs-Policy dokumentiert (in `docs/audit-retention.md`)
5. Optional: Alerting auf verdächtige Pattern (z.B. > 100 Schreib-Operationen pro Minute pro User)

## Effort

M — 1–3 Tage. Logger-Trennung + SIEM-Setup + Aufbewahrungs-Doku.
