# SEC-012 — Clickjacking-Protection

## Description

Wenn ein OAuth-Proxy-Server eine Consent-UI rendert (siehe SEC-001), ist diese UI ein lohnendes Ziel für Clickjacking: Ein Angreifer bettet die Consent-Seite in einem unsichtbaren `<iframe>` auf seiner eigenen Domain ein, überlagert sie mit harmlos aussehenden UI-Elementen («Klick hier für Gewinnspiel»), und der User klickt unwissentlich «Authorize» auf der echten Consent-Seite.

Mitigation in zwei Layern:

1. **`X-Frame-Options: DENY`** — alter, breit unterstützter Header. Verhindert dass die Seite in irgendeinem Frame geladen wird.

2. **CSP `frame-ancestors 'none'`** — moderner, expressiver Mechanismus. Erlaubt feinere Kontrolle (`'self'`, spezifische Origins). Hat Vorrang vor `X-Frame-Options`, wenn beide gesetzt sind.

Best Practice 2026: **beide setzen** als Defense-in-Depth, mit CSP als primärem Mechanismus. Der CSP-Header schützt zudem gegen weitere Angriffsklassen (XSS, Inline-Scripts), wenn richtig konfiguriert.

Dieser Check ist eng verwandt mit SEC-001 (Confused Deputy), der den Clickjacking-Schutz als eine der vier Anforderungen nennt. SEC-012 isoliert diesen Aspekt für eigenständiges Tracking, weil Server-Maintainer hier oft Lücken haben — gerade auf «interner» Seite zwischen Consent-UI und Callback.

## Verification

### Modus 1: code_review (Header gesetzt)

```bash
grep -rE 'X-Frame-Options|frame-ancestors' src/
grep -rE 'Content-Security-Policy|CSP' src/
```

**Pass-Pattern (Starlette / FastAPI Middleware):**

```python
from starlette.middleware.base import BaseHTTPMiddleware

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        # Auf allen Auth-Routen Clickjacking-Defense
        if request.url.path.startswith("/oauth/"):
            response.headers["X-Frame-Options"] = "DENY"
            response.headers["Content-Security-Policy"] = (
                "frame-ancestors 'none'; "
                "default-src 'self'; "
                "script-src 'self' 'unsafe-inline'; "  # nur falls nötig
                "style-src 'self' 'unsafe-inline'; "
                "img-src 'self' data:; "
                "form-action 'self';"
            )
            response.headers["Referrer-Policy"] = "no-referrer"
        return response

app.add_middleware(SecurityHeadersMiddleware)
```

**Pass-Pattern (FastMCP mit Custom Routes):**

```python
@app.get("/oauth/consent", response_class=HTMLResponse)
async def consent_page(request: Request, ...):
    html = render_consent_template(...)
    return HTMLResponse(
        content=html,
        headers={
            "X-Frame-Options": "DENY",
            "Content-Security-Policy": "frame-ancestors 'none'",
            "X-Content-Type-Options": "nosniff",
        },
    )
```

**Fail-Pattern:**

```python
# FAIL: Nur X-Frame-Options, kein CSP
response.headers["X-Frame-Options"] = "DENY"

# FAIL: CSP zu permissiv für Auth-UI
response.headers["Content-Security-Policy"] = "frame-ancestors *"

# FAIL: Header gesetzt, aber nicht auf OAuth-Routen
# (z.B. nur global mit Default-Werten)
```

### Modus 2: runtime_test (Header verifizieren)

```bash
# Consent-Seite auf Header prüfen
curl -i https://server/oauth/consent?client_id=test 2>/dev/null \
  | grep -iE 'X-Frame-Options|Content-Security-Policy'

# Erwartet:
# X-Frame-Options: DENY
# Content-Security-Policy: frame-ancestors 'none'; ...
```

### Modus 3: code_review (Auch interne Routen geschützt)

Häufiger Fehler: Consent-Seite ist geschützt, aber die Callback-Route (`/oauth/callback`) oder Token-Refresh-Route nicht. Bei manchen Clickjacking-Varianten reicht das Klicken auf eine Submit-Button-Imitation auf der Callback-Page.

```bash
# Welche Routen rendern HTML?
grep -rE 'HTMLResponse|render_template|@app\.get' src/ | grep -v 'JSONResponse'
```

**Pass:** Alle HTML-rendering-Routen im Auth-Bereich haben die Header.

## Pass Criteria

- [ ] `X-Frame-Options: DENY` für Consent-UI gesetzt
- [ ] `Content-Security-Policy: frame-ancestors 'none'` für Consent-UI gesetzt
- [ ] Beide Header werden auf allen HTML-Auth-Routen gesetzt (Consent, Callback, Logout)
- [ ] Zusätzlich: `Referrer-Policy: no-referrer` auf Auth-Routen
- [ ] Zusätzlich: `X-Content-Type-Options: nosniff` global
- [ ] Test verifiziert Header-Anwesenheit auf allen relevanten Routen

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Nur `X-Frame-Options`, kein CSP | Moderne Browser-Bypass-Tricks (z.B. via `<embed>` oder neue HTML5-Elemente) |
| CSP `frame-ancestors *` oder `'self' *` | Kein Schutz |
| Header global gesetzt, aber Auth-Routen nicht abgedeckt | Lücke bei spezifischen Auth-Pages |
| Header in HTML-Meta-Tag statt HTTP-Header | `frame-ancestors` MUSS als HTTP-Header sein, Meta-Tag wird ignoriert |
| `Referrer-Policy` fehlt | OAuth-State leak via Referer-Header |

## Remediation

### Schritt 1: Security-Headers-Middleware

Wie im Pass-Pattern. Als globale Middleware anwendbar, mit Path-spezifischen Verschärfungen für Auth-Routen.

### Schritt 2: Pro-Route-Verifikation

```python
# Liste aller HTML-rendering-Endpunkte
HTML_ROUTES = [
    "/oauth/consent",
    "/oauth/callback",
    "/oauth/logout",
    "/error",
]

@pytest.mark.parametrize("route", HTML_ROUTES)
def test_clickjacking_headers(client, route):
    response = client.get(route)
    assert response.headers.get("X-Frame-Options") == "DENY"
    csp = response.headers.get("Content-Security-Policy", "")
    assert "frame-ancestors 'none'" in csp
```

### Schritt 3: CI-Test gegen Regression

```yaml
- name: Check security headers
  run: pytest tests/test_security_headers.py -v
```

### Schritt 4: Bei API-only-Endpunkten

JSON-API-Routen (`/mcp/...`) brauchen keine Clickjacking-Headers, weil sie kein HTML rendern. Trotzdem schadet `Content-Type: application/json` + `X-Content-Type-Options: nosniff` nicht.
