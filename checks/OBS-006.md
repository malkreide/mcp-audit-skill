# OBS-006 — OpenTelemetry Tracing

## Description

OBS-005 deckt Audit-Logs für SIEM-Integration ab — Security-fokussiert. OBS-006 ergänzt das auf der **Performance- und Behavior-Seite**: jeder Tool-Call wird als OpenTelemetry-Span erfasst, mit:

- **Trace-ID** über die ganze LLM-Host → Gateway → Server → Backend-Kette
- **Tool-Name** als Span-Name
- **User-Identity** als Span-Attribut (aus `ctx.user_claims`)
- **Latenz** der einzelnen Hop-Schritte
- **Token-Count** falls Sampling involviert
- **Backend-API-Latenzen** als Child-Spans

Dieser Check ist `medium`, weil ohne Tracing zwar der Server funktioniert, aber drei wichtige Forensik- und Optimization-Workflows nicht möglich sind:

1. **Slow-Tool-Identifikation:** welche Tools haben P99-Latenzen > 5s?
2. **User-Behavior-Analyse:** welche Tool-Sequenzen sind anomal? (z.B. read+exfiltrate-Pattern)
3. **Backend-Bottleneck-Erkennung:** liegt die Latenz im Server oder bei der Datenquelle?

OpenTelemetry ist Industriestandard, vendor-neutral, mit Anbindern an alle gängigen Observability-Tools (Datadog, Splunk, Grafana Tempo, Honeycomb, Jaeger).

Im lokalen stdio-Setup ist Tracing weniger wichtig (Single-User-Latenzen sind meist nicht das Problem), daher `applies_when: deployment != "local-stdio"`.

## Verification

### Modus 1: code_review (OpenTelemetry-Setup)

```bash
grep -rE 'opentelemetry|otel|tracer\.start_as_current_span' src/
grep -rE 'opentelemetry-' pyproject.toml requirements.txt
```

**Pass-Pattern:**

```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor

# Setup beim Server-Start
provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter()))
trace.set_tracer_provider(provider)
tracer = trace.get_tracer(__name__)

# Auto-Instrumentation für ausgehende HTTP-Requests
HTTPXClientInstrumentor().instrument()


def traced_tool(tool_func):
    """Decorator: jeder Tool-Call wird zu einem OpenTelemetry-Span."""
    @functools.wraps(tool_func)
    async def wrapper(args, ctx):
        with tracer.start_as_current_span(
            f"mcp.tool.{tool_func.__name__}",
            attributes={
                "mcp.tool.name": tool_func.__name__,
                "mcp.user.id": ctx.user_claims.get("sub", "unknown"),
                "mcp.user.groups": ",".join(ctx.user_claims.get("groups", [])),
            },
        ) as span:
            try:
                result = await tool_func(args, ctx)
                span.set_attribute("mcp.tool.result.is_error", bool(result.get("isError")))
                return result
            except Exception as exc:
                span.record_exception(exc)
                span.set_status(trace.StatusCode.ERROR, str(exc))
                raise
    return wrapper


@traced_tool
@mcp.tool()
async def search_motions(args, ctx):
    return await db.search(args.query)
```

### Modus 3: code_review (sensitive Daten nicht in Spans)

Spans gehen in Observability-Backends. Sie dürfen **kein** PII oder Credential enthalten — auch nicht in Span-Attributen oder Stacktraces.

```bash
# Span-Attribute auf sensitive Daten prüfen
grep -rE 'set_attribute|span\.attributes' src/ | grep -iE 'password|token|secret|email|name'
```

**Fail-Pattern:**

```python
# FAIL: User-Email als Span-Attribut
span.set_attribute("mcp.user.email", ctx.user_claims["email"])

# FAIL: Tool-Args ungeprüft als Attribut
span.set_attribute("mcp.tool.args", json.dumps(args.dict()))
```

**Pass-Pattern:**

```python
# OK: User-ID (opaker Identifier), keine PII
span.set_attribute("mcp.user.id", ctx.user_claims.get("sub"))

# OK: Tool-Name, kein Args-Inhalt
span.set_attribute("mcp.tool.name", tool_func.__name__)

# OK: Args-Hash (für Korrelation, aber kein Inhalt)
span.set_attribute("mcp.tool.args_hash", hashlib.sha256(str(args).encode()).hexdigest()[:16])
```

## Pass Criteria

- [ ] OpenTelemetry SDK installiert und initialisiert
- [ ] TracerProvider konfiguriert mit OTLP-Exporter
- [ ] Auto-Instrumentation für HTTP-Clients (httpx, requests) aktiv
- [ ] Pro Tool-Call ein Span mit: `mcp.tool.name`, `mcp.user.id`, `mcp.tool.result.is_error`
- [ ] Backend-API-Calls automatisch als Child-Spans (via Auto-Instrumentation)
- [ ] **Keine** sensitiven Daten in Span-Attributen (kein PII, kein Token, kein freier Args-Inhalt)
- [ ] OTLP-Endpoint via Env-Var konfigurierbar (typisch in Cloud-Deployments)
- [ ] Service-Name und Environment-Tag explizit gesetzt

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Kein Tracing — nur Logs | Forensik / Performance-Analyse stark eingeschränkt |
| Tracing ohne Auto-Instrumentation für HTTP | Backend-Bottlenecks unsichtbar |
| Span-Attribute mit User-Email / Token | PII-Leak ins Observability-Backend |
| OTLP-Endpoint hardcoded (z.B. localhost:4317) | Bricht in Production |
| Keine Service-Name-Setzung | Spans nicht filtrierbar pro Server |

## Remediation

### Schritt 1: SDK-Installation

```toml
# pyproject.toml
[project.dependencies]
"opentelemetry-api" = "^1.21"
"opentelemetry-sdk" = "^1.21"
"opentelemetry-exporter-otlp" = "^1.21"
"opentelemetry-instrumentation-httpx" = "^0.42b0"
```

### Schritt 2: Setup-Modul

```python
# src/server_name/observability.py
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.resources import Resource
# ...

def setup_tracing():
    resource = Resource.create({
        "service.name": os.environ.get("OTEL_SERVICE_NAME", "schulamt-mcp"),
        "deployment.environment": os.environ.get("ENVIRONMENT", "development"),
    })
    provider = TracerProvider(resource=resource)
    provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter()))
    trace.set_tracer_provider(provider)
    HTTPXClientInstrumentor().instrument()
```

### Schritt 3: Decorator anwenden

`@traced_tool` als Standard auf alle Tool-Decorators stacken.

### Schritt 4: OTLP-Backend wählen

Für Schulamt-Kontext: Datadog (DSG-konform mit `DD_SITE=datadoghq.eu`), Grafana Tempo (selbst-gehostet, OpenBao-Compatible), oder Honeycomb (EU-Region).

## Effort

M — 1–3 Tage. SDK-Setup + Decorator + Backend-Konfiguration + Tests.
