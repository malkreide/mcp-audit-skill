# SDK-001 — FastMCP Lifespan Management

## Description

MCP-Server halten häufig Ressourcen, die über die einzelne Tool-Anfrage hinaus existieren: HTTP-Connection-Pools, DB-Pools, Redis-Verbindungen, gecachte Auth-Tokens, Pre-Computed-Indexes. Werden diese pro Tool-Call neu erzeugt, bricht die Performance ein. Werden sie gar nicht aufgeräumt, ergeben sich Resource-Leaks (offene TCP-Connections, dangling Cursor).

FastMCP bietet das Lifespan-Pattern dafür: Eine `@asynccontextmanager`-Funktion erhält den FastMCP-Server, initialisiert Ressourcen vor dem ersten Request und räumt sie nach dem letzten Request sauber ab. Im Multi-Server-Setup (mehrere MCPs in einer ASGI-App) wird `AsyncExitStack` zum kollektiven Lifecycle-Management.

## Verification

### Modus 1: code_review (Lifespan-Funktion vorhanden)

```bash
grep -rE "@asynccontextmanager|@contextlib\.asynccontextmanager" src/
grep -rE "FastMCP\([^)]*lifespan=" src/
```

**Pass-Pattern:**

```python
from contextlib import asynccontextmanager
from fastmcp import FastMCP
import httpx
from redis.asyncio import Redis

@asynccontextmanager
async def lifespan(server: FastMCP):
    server.state.http = httpx.AsyncClient(timeout=30, follow_redirects=False)
    server.state.redis = Redis.from_url(os.environ["REDIS_URL"])
    try:
        yield
    finally:
        await server.state.http.aclose()
        await server.state.redis.aclose()

mcp = FastMCP("zurich-opendata", lifespan=lifespan)

@mcp.tool()
async def search(query: str, ctx):
    # Resource-Reuse, nicht pro Call neu
    response = await ctx.fastmcp.state.http.get(f"https://api/search?q={query}")
    return response.json()
```

### Modus 2: code_review (AsyncExitStack bei Multi-Server)

Falls mehrere MCPs in einer ASGI-App gemounted sind:

```python
from contextlib import AsyncExitStack
from starlette.applications import Starlette
from starlette.routing import Mount

@asynccontextmanager
async def combined_lifespan(app):
    async with AsyncExitStack() as stack:
        await stack.enter_async_context(mcp_a.session_manager.run())
        await stack.enter_async_context(mcp_b.session_manager.run())
        yield

app = Starlette(
    routes=[
        Mount("/a", app=mcp_a.streamable_http_app()),
        Mount("/b", app=mcp_b.streamable_http_app()),
    ],
    lifespan=combined_lifespan,
)
```

**Fail-Pattern (Resource-Leak):**

```python
mcp = FastMCP("server")

@mcp.tool()
async def search(query: str):
    # Neuer HTTP-Client pro Aufruf — kein Connection-Pooling, Leaks bei Fehler
    async with httpx.AsyncClient() as client:
        return await client.get(...)
```

## Pass Criteria

- [ ] Lifespan-Funktion mit `@asynccontextmanager` definiert
- [ ] FastMCP-Konstruktor erhält `lifespan=lifespan`
- [ ] Cleanup im `finally`-Block oder via Context-Manager-Exit (`async with`)
- [ ] Bei Multi-Server-Setup: `AsyncExitStack` für kollektives Lifecycle-Management
- [ ] Keine `httpx.AsyncClient()` / `Redis()` / `asyncpg.connect()` pro Tool-Call

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| HTTP-Client pro Call neu erstellt | Kein Connection-Pooling, langsam, Leak-Gefahr |
| Cleanup nicht im `finally` | Bei Exception bleiben Ressourcen offen |
| Globale Module-Variablen statt `server.state` | Schlecht testbar, Race Conditions möglich |
| `__aenter__` / `__aexit__` selbst implementieren statt `@asynccontextmanager` | Mehr Boilerplate, fehleranfälliger |

## Remediation

Migrationsweg:

```diff
+ from contextlib import asynccontextmanager
+ import httpx
+
+ @asynccontextmanager
+ async def lifespan(server):
+     server.state.http = httpx.AsyncClient(timeout=30)
+     try:
+         yield
+     finally:
+         await server.state.http.aclose()
+
- mcp = FastMCP("zurich-opendata")
+ mcp = FastMCP("zurich-opendata", lifespan=lifespan)

  @mcp.tool()
- async def search(query: str):
-     async with httpx.AsyncClient() as client:
-         return (await client.get(f"https://api/{query}")).json()
+ async def search(query: str, ctx):
+     return (await ctx.fastmcp.state.http.get(f"https://api/{query}")).json()
```
