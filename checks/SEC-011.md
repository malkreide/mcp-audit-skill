# SEC-011 — Cookie-Security

## Description

Cookies sind das primäre Vehikel für Session-State im OAuth-Proxy-Pattern (Consent-Flow, State-Parameter, Session-Tokens). Falsch konfigurierte Cookies öffnen sich für vier verschiedene Angriffsklassen:

| Angriff | Mitigation | Cookie-Attribut |
|---|---|---|
| **Plaintext-MITM** | TLS erzwingen | `Secure` |
| **JS-Diebstahl (XSS)** | JS-Zugriff blocken | `HttpOnly` |
| **CSRF / Cross-Site-Request** | Cross-Origin-Send blocken | `SameSite=Strict` oder `Lax` |
| **Subdomain-Takeover** | Cookie strikt an Origin binden | `__Host-` Prefix |

Der `__Host-` Prefix ist 2026 nicht mehr optional, sondern Pflicht für Auth-Cookies. Er erzwingt drei Eigenschaften gleichzeitig:

1. `Secure` muss gesetzt sein
2. `Path=/` (kein Path-Restriction-Trick)
3. `Domain` darf NICHT gesetzt sein → Cookie ist strikt an die Origin gebunden, keine Subdomain-Vererbung

Damit kann ein kompromittiertes `evil.example.com` keine Cookies setzen, die `app.example.com` lesen würde — selbst wenn sie via DNS-Wildcard auf demselben Server landen.

## Verification

### Modus 1: code_review (Cookie-Setzungen)

```bash
grep -rE 'set_cookie|setCookie|Set-Cookie' src/
```

**Pass-Pattern (FastAPI / Starlette):**

```python
from starlette.responses import Response

response = Response()
response.set_cookie(
    key="__Host-mcp-session",     # Pflicht-Prefix für Auth-Cookies
    value=signed_token,
    secure=True,                   # nur HTTPS
    httponly=True,                 # kein JS-Zugriff
    samesite="strict",             # kein Cross-Origin-Send
    path="/",                      # zwingend bei __Host-
    # KEIN domain= — damit __Host- gilt
    max_age=3600,                  # explizite TTL
)
```

**Fail-Patterns:**

```python
# FAIL 1: Kein Prefix, kein Secure
response.set_cookie("session", token)

# FAIL 2: Domain gesetzt — bricht __Host-Constraint
response.set_cookie(
    "__Host-session", token,
    secure=True, httponly=True,
    domain=".example.com",  # ← __Host- fordert KEIN domain
)

# FAIL 3: SameSite=None ohne explizite Begründung
response.set_cookie(
    "session", token,
    secure=True, samesite="none",  # ← öffnet CSRF-Angriffsfläche
)

# FAIL 4: Keine TTL → Cookie wird Session-Cookie, lebt bis Browser-Close
response.set_cookie("session", token, secure=True, httponly=True)
```

### Modus 2: runtime_test (Cookie-Header inspect)

```bash
curl -i -X POST https://server/oauth/callback?code=...\&state=... \
  | grep -i "set-cookie:"
# Erwartet: alle Auth-Cookies mit __Host-, Secure, HttpOnly, SameSite=Strict|Lax
```

### Modus 3: code_review (SameSite-Wert begründet)

`SameSite=Strict` ist defensiv-strikt: Cookie wird nicht einmal beim Klick auf einen Link von einer anderen Origin gesendet. Das bricht typische OAuth-Redirect-Flows. `SameSite=Lax` ist der Pragmatik-Standard für OAuth: Cookie wird bei Top-Level-Navigation gesendet, nicht bei Iframe / XHR.

```bash
grep -rE 'samesite' src/
```

**Pass:** Wert ist `strict` oder `lax`, nicht `none`. Falls `none`: Begründung dokumentiert.

### Modus 4: code_review (TTL explizit)

Auth-Cookies sollen nicht ewig leben. TTL korreliert mit Session-Lifetime (siehe SEC-009).

```bash
grep -rE 'max_age|maxAge|expires' src/
```

**Pass:** `max_age` ist explizit gesetzt, < 24h für Standard-Sessions, < 10min für OAuth-State (siehe SEC-010).

## Pass Criteria

- [ ] Auth-Cookies haben `__Host-` Prefix
- [ ] `Secure=True` für alle Auth-Cookies
- [ ] `HttpOnly=True` für alle Auth-Cookies
- [ ] `SameSite=Strict` (bevorzugt) oder `Lax` für alle Auth-Cookies
- [ ] **Kein** `domain=`-Attribut bei `__Host-` Cookies
- [ ] `path=/` explizit bei `__Host-` Cookies
- [ ] `max_age` explizit gesetzt mit sinnvoller TTL
- [ ] Cookies sind kryptografisch signiert (HMAC oder JWT) — Inhalt nicht client-manipulierbar

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Cookie ohne Prefix | Subdomain-Takeover-Vector |
| `__Host-` mit `domain=` | Browser ignoriert das `__Host-` und behandelt es als normales Cookie |
| `SameSite=None` ohne MFA-Defense | CSRF-Angriffsfläche |
| Kein `max_age` | Cookie lebt bis Browser-Restart, theoretisch unbegrenzt |
| Unsignierter Cookie-Wert | Client kann Inhalt manipulieren |

## Remediation

### Schritt 1: Cookie-Helper zentralisieren

```python
from itsdangerous import URLSafeTimedSerializer
from starlette.responses import Response

signer = URLSafeTimedSerializer(SECRET_KEY)

def set_auth_cookie(response: Response, key: str, value: dict, max_age: int):
    """Standardisiertes Cookie-Setzen für alle Auth-Flows."""
    signed = signer.dumps(value)
    response.set_cookie(
        key=f"__Host-{key}",
        value=signed,
        secure=True,
        httponly=True,
        samesite="lax",  # oder "strict" für höhere Sicherheit
        path="/",
        max_age=max_age,
        # explizit: domain NICHT setzen
    )

def read_auth_cookie(request, key: str, max_age: int) -> dict | None:
    raw = request.cookies.get(f"__Host-{key}")
    if not raw:
        return None
    try:
        return signer.loads(raw, max_age=max_age)
    except (BadSignature, SignatureExpired):
        return None
```

### Schritt 2: Alle bestehenden Cookie-Sites refaktorieren

Suche im Code:

```bash
grep -rn "set_cookie" src/ | grep -v "set_auth_cookie"
```

Jede Stelle prüfen und auf den Helper umstellen.

### Schritt 3: Tests gegen Cookie-Header

```python
def test_auth_cookie_has_required_attributes(client):
    response = client.post("/oauth/login", ...)
    cookie_header = response.headers.get("Set-Cookie", "")

    assert "__Host-" in cookie_header
    assert "Secure" in cookie_header
    assert "HttpOnly" in cookie_header
    assert "SameSite=" in cookie_header
    assert "Domain=" not in cookie_header  # __Host- darf kein Domain
    assert "Max-Age=" in cookie_header
```
