# CH-008 — EDÖB-Meldepflicht bei Datenschutz-Verletzungen

## Description

Das revidierte DSG (Art. 24) verpflichtet Verantwortliche, **Verletzungen der Datensicherheit** dem EDÖB (Eidgenössischer Datenschutz- und Öffentlichkeitsbeauftragter) so rasch als möglich zu melden, sofern die Verletzung voraussichtlich zu einem hohen Risiko für die Persönlichkeit oder Grundrechte der betroffenen Personen führt. Die Meldung muss enthalten:

- Art der Verletzung
- Folgen
- Bereits ergriffene Massnahmen

Zusätzlich ist die betroffene Person zu informieren, wenn das zu ihrem Schutz erforderlich ist oder der EDÖB es verlangt.

Für einen MCP-Server bedeutet das:

1. **Detection**: Mechanismen, um Verletzungen überhaupt zu erkennen (Monitoring, Anomaly Detection, Alert auf Zugriffs-Pattern)
2. **Triage**: Klare Eskalations-Kette wer was wann meldet
3. **Documentation**: Incident-Response-Plan im Repo, mit konkreten Schritten
4. **Practice**: Mindestens einmal pro Jahr Tabletop-Übung

Beispiele für meldepflichtige Verletzungen im Schulamts-Kontext:
- Datenleck einer Schüler:innen-Liste über GitHub-Public-Repo
- Unauthorized Access auf Lehrer:innen-PII via fehlerhafte Auth
- API-Key-Leak führt zu Drittlandtransfer (LLM-Provider) mit PII
- Erfolgreicher SSRF-Angriff (siehe SEC-004) mit Daten-Exfiltration

Dieser Check ist `critical`, weil ohne dokumentierte Incident-Response der Maintainer im Ernstfall improvisieren muss — und Meldefristen verstreichen lässt.

## Verification

### Modus 1: documentation_check (Incident-Response-Plan)

```bash
find . -iname '*incident*' -o -iname '*breach*' -o -iname '*security-response*' -o -iname '*runbook*'
```

**Pass-Pattern (`docs/incident-response.md`):**

```markdown
# Incident Response Plan: <server-name>

## Eskalations-Kette

| Rolle | Person | Kontakt | Erreichbarkeit |
|---|---|---|---|
| Server-Maintainer | <Hayal> | hayal@... | Mo-Fr 8-17h |
| Datenschutzbeauftragte:r Stadt Zürich | <Name> | datenschutz@zuerich.ch | werktags |
| OIZ Sicherheit | OIZ-CSIRT | <Hotline> | 24/7 |
| EDÖB (extern) | — | https://www.edoeb.admin.ch/ | werktags |

## Detection

- Monitoring auf folgende Pattern (Datadog/Splunk-Alerts):
  - > 100 Failed-Auth pro Minute
  - Unerwartete 4xx/5xx-Spikes
  - Zugriffe ausserhalb Schweiz/EU IP-Range (falls geo-restricted)
  - Audit-Log-Anomalien (z.B. plötzliche Bulk-Reads)
- Wöchentliche Container-CVE-Scans (Trivy, Snyk)
- Monatliche Code-Reviews der kritischen Pfade

## Triage-Workflow

1. **Erkennung** (durch Maintainer, OIZ-CSIRT, externer Hinweis)
2. **Sofort-Containment** — Server in Wartungsmodus, Tokens rotieren
3. **Bewertung** (innerhalb 24h):
   - Welche Daten betroffen?
   - Wie viele Personen betroffen?
   - Wahrscheinlich «hohes Risiko» nach revDSG Art. 24?
4. **Meldung** (innerhalb 72h, falls hohes Risiko):
   - EDÖB via `https://databreach.edoeb.admin.ch/`
   - Datenschutzbeauftragte:r Stadt Zürich
   - Bei Lehrpersonen-Daten: ggf. ZLV/LCH (siehe CH-003)
5. **Information** der betroffenen Personen (falls verlangt)
6. **Forensik** — was war die Root Cause?
7. **Remediation** — Code-Fix, Pattern-Update im Audit-Skill
8. **Post-Mortem** (innerhalb 2 Wochen)

## Vorlage EDÖB-Meldung

Felder gemäss Art. 24 revDSG:
- Art der Verletzung: <z.B. "Unauthorized Access auf Tool-Endpoint">
- Zeitpunkt: <Datum + UTC-Zeit>
- Betroffene Personen: <Anzahl + Kategorien>
- Betroffene Datentypen: <Liste>
- Folgen / Risiken: <Bewertung>
- Bereits ergriffene Massnahmen: <Liste>
- Geplante weitere Massnahmen: <Liste>
- Kontakt für Rückfragen: <Person>

## Übungs-Plan

- Jährliche Tabletop-Übung (Q3)
- Ad-hoc nach jedem Live-Incident
```

**Pass:** Mindestens ein dokumentierter Übungs-Durchlauf in den letzten 12 Monaten.

## Pass Criteria

- [ ] Incident-Response-Plan im Repo (`docs/incident-response.md`)
- [ ] Eskalations-Kette mit konkreten Personen + Kontakten
- [ ] Detection-Mechanismen aktiv (Monitoring, Alerts)
- [ ] EDÖB-Meldungs-Vorlage vorhanden
- [ ] Triage-Workflow mit klaren Zeitfristen (72h für EDÖB)
- [ ] Mindestens eine Tabletop-Übung in den letzten 12 Monaten

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Kein Incident-Response-Plan | Improvisation im Ernstfall, verstrichene Fristen |
| Plan vorhanden, aber nie getestet | Kritische Lücken werden erst im Ernstfall sichtbar |
| Keine Detection — Verletzung wird gar nicht bemerkt | DSG-Verstoss eskaliert |
| EDÖB-Meldung in plain Mail statt offizielles Portal | Formalfehler |
| Verzögerung > 72h ohne Begründung | Verstoss gegen Meldefrist |

## Remediation

### Schritt 1: Plan erstellen

Template aus dem Pass-Pattern oben. Mit Maintainer + Datenschutzbeauftragte:r durchgehen, konkrete Personen eintragen.

### Schritt 2: Detection-Setup

Mindestens: Datadog/Splunk-Alerts auf Failed-Auth-Spikes und 5xx-Error-Rate.

### Schritt 3: Tabletop-Übung

In ca. 1 Stunde durchspielbar:

> «Stell dir vor, ein Issue auf GitHub meldet, dass `zh-education-mcp` Schüler-Namen statt aggregierte Statistiken zurückgibt. Welche Schritte unternimmst du in den nächsten 72h?»

Durchspielen, Lücken im Plan identifizieren, Plan aktualisieren.

### Schritt 4: Jährlicher Review

Plan, Eskalations-Kette und Übungs-Resultate jährlich aktualisieren.

## Effort

M — 2–3 Tage für Plan-Erstellung + Setup. Übungs-Durchlauf ca. 1h.
