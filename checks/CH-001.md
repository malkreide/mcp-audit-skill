# CH-001 — DSG/EDÖB: Datenresidenz Schweiz/EU

## Description

Das revidierte Schweizer Datenschutzgesetz (revDSG, in Kraft seit 1.9.2023) und die EDÖB-Empfehlungen verlangen, dass Personendaten und Verwaltungsdaten der öffentlichen Hand grundsätzlich auf Servern in der Schweiz oder einem Land mit angemessenem Datenschutzniveau (EU/EWR-Staaten + Liste laut Anhang 1 DSV) verarbeitet werden. Bei Übermittlung in Drittländer (insbesondere USA, Asien) sind zusätzliche Garantien erforderlich (Standardvertragsklauseln, BCRs, oder explizite Einwilligung).

Für MCP-Server der Stadtverwaltung Zürich gilt zusätzlich die ISDS-Richtlinie und die Klassifikationspflicht. Ein MCP-Server, der Verwaltungsdaten oder Personendaten verarbeitet **und** in einem Drittland gehostet wird, ist ohne weitere Massnahmen DSG-widrig.

## Verification

### Modus 1: config_check (Hosting-Region)

```bash
# Wo läuft der Server?
# Railway: Region in railway.toml oder Dashboard
cat railway.toml 2>/dev/null
# Render: Region in render.yaml
cat render.yaml 2>/dev/null
# Cloudflare: meist global (Edge-Network)
cat wrangler.toml 2>/dev/null
# Fly.io: primary_region
cat fly.toml 2>/dev/null
```

**Akzeptable Regionen:**
- Schweiz: ZRH (Zürich), GVA (Genf)
- EU/EWR: FRA (Frankfurt), AMS (Amsterdam), DUB (Dublin), STO (Stockholm), MIL (Mailand), MAD (Madrid), WAW (Warschau), etc.
- Andere mit Angemessenheitsbeschluss: GB (London), Liechtenstein, Norwegen, Island

**Nicht akzeptabel ohne Zusatzgarantien:**
- USA-Regionen (IAD, SFO, ORD, LAX, etc.) — auch wenn Anbieter EU-Garantien bietet, ist die juristische Lage komplex
- Asien-Regionen (HND, SIN, SYD, etc.)
- Edge-Computing mit globaler Replikation ohne Region-Sperre

### Modus 2: code_review (Logging-Senken)

Selbst bei korrekter Hosting-Region können Daten via Logging in Drittländer abfliessen.

```bash
# Suche nach externen Log-Senken
grep -rE "datadog|sentry|new relic|loggly|splunk" src/ pyproject.toml package.json
grep -rE "honeycomb|axiom|loki" src/ pyproject.toml package.json
```

**Pass-Pattern:**
- Sentry: EU-Region (`sentry.io` mit `*.eu.sentry.io` oder Self-Hosted)
- Datadog: EU-Region (`*.datadoghq.eu`, nicht `.com`)
- Strukturierte Logs in eigenem Bucket (S3 EU, GCS EU)

**Fail-Pattern:**
- Default Sentry-DSN ohne Region-Spezifikation (geht standardmässig in USA)
- Datadog ohne `DD_SITE=datadoghq.eu`
- Console.log + Cloud-Anbieter mit USA-Default-Region

### Modus 3: code_review (Drittanbieter-API-Calls)

Werden Personen- oder Verwaltungsdaten an externe APIs weitergeleitet?

```bash
# Suche nach externen API-Hosts
grep -rE "openai\.com|anthropic\.com|cohere\.com|huggingface\.co" src/
grep -rE "googleapis\.com|microsoftonline\.com|azure\.com" src/
```

**Bewertung pro Drittanbieter:**

| Anbieter | DSG-Status | Notwendige Massnahmen |
|---|---|---|
| OpenAI (USA) | Drittland | Auftragsdatenverarbeitungsvertrag, ggf. Standardvertragsklauseln, Einwilligung |
| Anthropic (USA) | Drittland | Wie OpenAI |
| Anthropic (EU-Endpunkt, falls verfügbar) | EU | Standardpraxis |
| Google Cloud (EU-Region) | EU | Falls EU-Region und EU-Garantien aktiviert: ok |
| Microsoft Azure (Schweiz Nord/West) | CH | Datenresidenz-Add-on aktivieren |

### Modus 4: documentation_check (Verarbeitungsverzeichnis)

DSG verlangt für Verwaltungsdaten ein Verarbeitungsverzeichnis. Im Repo prüfen:

```bash
find . -iname "*verarbeitung*" -o -iname "*processing*" -o -iname "*data-flow*" -o -iname "*dpia*"
ls docs/ -la 2>/dev/null
```

**Pass:** Verarbeitungsverzeichnis ist im Repo, im Wiki, oder in einer verlinkten Dokumentation vorhanden mit:
- Zweck der Verarbeitung
- Kategorien betroffener Personen
- Kategorien Personendaten
- Datenempfänger (intern/extern)
- Aufbewahrungsfrist
- Löschkonzept

## Pass Criteria

- [ ] Hosting in Schweiz oder EU/EWR/Liste-Anhang-1-DSV
- [ ] Logging-Senken in CH/EU oder Self-Hosted
- [ ] Bei Drittanbieter-API-Calls: dokumentierter Auftragsdatenverarbeitungsvertrag oder Datenminimierung (keine PII übertragen)
- [ ] Bei Datenklasse `Verwaltungsdaten` oder `PII`: Verarbeitungsverzeichnis vorhanden
- [ ] Bei Datenklasse `PII`: explizit dokumentierte Rechtsgrundlage (Einwilligung, Vertragserfüllung, gesetzliche Pflicht)

## Common Failures

| Pattern | Risiko |
|---|---|
| Server auf Railway USA-East gehostet | DSG-Verstoss bei Verwaltungsdaten |
| Sentry mit Default-DSN (USA) | Fehler-Logs mit Personendaten in USA |
| `console.log(user_data)` mit Container-Logs in USA-Region | gleiches Problem |
| Direkter API-Call zu OpenAI mit Personendaten | DSG-Verstoss ohne ZV/SCC |
| Kein Verarbeitungsverzeichnis | DSG-Pflichtverletzung, EDÖB-relevant |

## Remediation

### Schritt 1: Hosting-Region verifizieren / migrieren

```toml
# railway.toml — auf EU-Region setzen
[build]
builder = "nixpacks"

[deploy]
region = "europe-west4"  # Frankfurt oder Amsterdam
```

```yaml
# render.yaml
services:
  - type: web
    name: my-mcp
    region: frankfurt  # nicht oregon/ohio
```

### Schritt 2: Logging-Senken auf EU umstellen

```python
# Sentry mit EU-DSN
sentry_sdk.init(
    dsn="https://xxx@oXXXXXX.ingest.eu.sentry.io/yyy",  # .eu.sentry.io
    environment="production",
)

# Datadog
os.environ["DD_SITE"] = "datadoghq.eu"  # nicht datadoghq.com
```

### Schritt 3: Datenminimierung vor externen API-Calls

```python
def redact_pii(text: str) -> str:
    """Entfernt Personendaten vor LLM-Übermittlung."""
    text = re.sub(r"\b[\w.-]+@[\w.-]+\.\w+\b", "[EMAIL]", text)
    text = re.sub(r"\b\d{3}\s?\d{3}\s?\d{2}\s?\d{2}\b", "[PHONE]", text)
    text = re.sub(r"\b756\.\d{4}\.\d{4}\.\d{2}\b", "[AHV]", text)  # AHV-Nr CH
    return text
```

### Schritt 4: Verarbeitungsverzeichnis als `docs/dsg-processing-record.md`

Template:

```markdown
# Verarbeitungsverzeichnis: <server-name>

## Zweck der Bearbeitung
<konkret, z.B. "Suche in städtischen Bildungsstatistiken">

## Rechtsgrundlage
- [ ] Einwilligung (Art. 31 Abs. 1 DSG)
- [ ] Vertragserfüllung (Art. 31 Abs. 2 lit. a DSG)
- [ ] Gesetzliche Pflicht (Art. 31 Abs. 2 lit. e DSG)
- [ ] Öffentliches Interesse (Art. 31 Abs. 2 lit. e DSG)

## Kategorien Personendaten
- z.B. Schüler-IDs, Lehrer-Namen, Schul-Adressen

## Datenempfänger
- Intern: <Liste städtischer Stellen>
- Extern: <falls zutreffend, mit Rechtsgrundlage>

## Aufbewahrungsfrist
<konkret>

## Löschkonzept
<wie und wann>

## Standort der Verarbeitung
<Hosting-Region, Backup-Region>

## Sicherheitsmassnahmen
<Verschlüsselung, Zugriffskontrolle, etc.>
```

## Effort

L — 1–2 Wochen für Hosting-Migration + Logging-Anpassung + Verarbeitungsverzeichnis. Bei laufenden Production-Servern mit User-Daten: zusätzliche Zeit für saubere Migration.
