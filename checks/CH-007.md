# CH-007 — Datenresidenz Backup-Region

## Description

CH-001 deckt die Hosting-Region des Production-Servers ab. CH-007 erweitert das auf Backups: Backup-Daten zählen rechtlich als gleichwertige Datenverarbeitung. Wenn Production in Frankfurt läuft, aber Backups automatisch in eine USA-Region repliziert werden, ist die DSG-Konformität nicht hergestellt.

Typische Konstellationen mit verstecktem Drittland-Backup:
- AWS S3 Cross-Region Replication mit Default-Region us-east-1
- Datadog-Logs in eu-Region, aber Trace-Backups via Datadog-Internal in US
- Railway Production in EU, aber Build-Caches/Logs in US
- Cloudflare R2 Backup-Bucket ohne Region-Sperre

Der Best-Practice-Standard verlangt:

1. **Inventar aller Backup-Ziele** (DB-Backups, Log-Backups, Build-Artefakte, Container-Image-Registry)
2. **Region-Lock pro Ziel** explizit auf Schweiz oder EU-Liste-Anhang-1-DSV
3. **Dokumentation** im Verarbeitungsverzeichnis (Synergie zu CH-002)

## Verification

### Modus 1: config_check (Backup-Konfiguration)

```bash
# AWS S3 Replication
grep -rE 'replication|cross_region|cross-region' deploy/ k8s/

# Datenbank-Backup-Region
grep -rE 'backup_region|backup.*region|pgbackrest' deploy/ docker-compose.yml

# Container-Registry
grep -rE 'registry|ecr|gcr|ghcr' deploy/
```

### Modus 2: documentation_check (Backup-Inventar)

```bash
find . -iname '*backup*' -o -iname '*disaster-recovery*'
```

**Pass-Pattern (`docs/backup-inventory.md`):**

```markdown
# Backup-Inventar: <server-name>

| Backup-Typ | Ziel | Region | Aufbewahrung | Verschlüsselung |
|---|---|---|---|---|
| Database (PostgreSQL) | AWS S3 | eu-central-1 (Frankfurt) | 30 Tage | AES-256 at-rest, TLS in-transit |
| Log-Archive | Datadog | eu | 90 Tage | TLS, Datadog at-rest |
| Container Images | GitHub Container Registry | EU/US (CDN) | unbegrenzt | TLS |
| Build Artifacts | GitHub Actions | EU/US (CDN) | 30 Tage | — |

## Drittland-Risiken

- GitHub Container Registry: globaler CDN, nicht ausschliesslich EU.
  Risiko-Bewertung: niedrig, da Container-Images keine Personendaten enthalten.
- GitHub Actions Build-Artefakte: gleicher Status. Build-Logs werden auf
  PII-Pattern gefiltert.
```

### Modus 3: code_review (Verschlüsselung-as-Defense)

Wenn ein Backup-Ziel partiell in Drittland liegt (z.B. CDN-replizierter Container-Registry), kann Client-Side-Verschlüsselung mit in der EU verwaltetem Schlüssel die Datenresidenz herstellen — kryptografisch, nicht geografisch.

```bash
grep -rE 'kms|sops|age|client.side.encryption' deploy/
```

## Pass Criteria

- [ ] Backup-Inventar vorhanden mit allen Backup-Typen
- [ ] Pro Backup-Typ: Region explizit gesetzt und dokumentiert
- [ ] Production-Region und Backup-Region in DSG-konformer Geografie
- [ ] Bei partieller Drittland-Replikation: Risiko-Bewertung dokumentiert + Mitigation (Client-Side-Encryption)
- [ ] Aufbewahrungsfrist pro Backup-Typ explizit (DSG Art. 6 Datenminimierung)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Default-Region us-east-1 für S3-Backup | DSG-Verstoss |
| Datadog ohne `DD_SITE=datadoghq.eu` | Logs in USA |
| Backup-Aufbewahrung «unbegrenzt» | Verstoss gegen Datenminimierung |
| Container-Registry global ohne Encryption-at-Rest-Garantien | Risiko bei sensitiven Build-Artefakten |

## Remediation

### Schritt 1: Backup-Inventar erstellen

Pro Datenfluss (Production → Backup) durchgehen, Region prüfen, in Tabelle dokumentieren.

### Schritt 2: Region-Locks setzen

```yaml
# AWS S3 (Terraform-Beispiel)
resource "aws_s3_bucket" "backup" {
  bucket = "schulamt-mcp-backup"
  # Region wird durch Provider gesetzt
}

resource "aws_s3_bucket_replication_configuration" "backup_replication" {
  bucket = aws_s3_bucket.backup.id
  role   = aws_iam_role.replication.arn

  rule {
    id     = "eu-only"
    status = "Enabled"

    destination {
      bucket        = aws_s3_bucket.backup_secondary.arn  # ebenfalls EU-Region
      storage_class = "STANDARD_IA"
    }
  }
}
```

### Schritt 3: Bei Drittland-CDN: Client-Side-Encryption

```python
# Beispiel: SOPS für Secrets-Encryption vor Backup-Upload
# sops --kms arn:aws:kms:eu-central-1:...:key/... --encrypt secrets.yaml
```

## Effort

S — < 1 Tag für Inventar + Region-Locks. Bei Migration bestehender Backups: 1–3 Tage.
