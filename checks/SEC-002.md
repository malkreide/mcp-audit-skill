# SEC-002 — Token Passthrough Prohibition

## Description

«Token Passthrough» bezeichnet das unkritische Weiterreichen eines vom MCP-Client erhaltenen Access-Tokens an einen Upstream-Service (z.B. interne API, Drittanbieter-API). Auf den ersten Blick wirkt das pragmatisch — der Server muss kein eigenes Auth-Konzept implementieren —, ist aber ein Verstoss gegen die OAuth-2.0-Sicherheitsarchitektur.

**Konsequenzen:**

1. **Audit-Trail-Bruch:** Der Upstream-Service sieht den End-User, nicht den MCP-Server als Aufrufer. Forensik bei Vorfällen wird unmöglich.
2. **Umgangene Rate-Limits:** Der Upstream-Service kennt nur eine User-Identität, kann aber nicht zwischen direkten User-Calls und MCP-vermittelten Calls unterscheiden.
3. **Verletzte Vertrauensgrenzen:** Token, das für Service A ausgestellt wurde, wird in Service B verwendet. Klassischer «Confused Deputy» auf Token-Ebene.
4. **Privilege Escalation:** MCP-Server erhält faktisch alle Berechtigungen des Users gegenüber dem Upstream-Service.

RFC 8707 («Resource Indicators for OAuth 2.0») ist die Mitigation: Tokens werden gegen einen `audience`-Claim geprüft, der explizit den vorgesehenen Empfänger benennt. Ein Token mit falscher `audience` wird mit HTTP 401 abgelehnt.

## Verification

### Modus 1: code_review (Audience-Validation aktiv)

```bash
# Suche nach Audience-Validierung im Token-Verifizierungs-Code
grep -rE "audience|aud|verify.*aud" src/ --include="*.py" --include="*.ts"
grep -rE "RFC.*8707|resource_indicator" src/
```

**Pass-Pattern (Python, FastMCP + Authlib):**

```python
from authlib.jose import jwt
from authlib.jose.errors import JoseError

EXPECTED_AUDIENCE = "https://mcp.schulamt-zuerich.ch"

def verify_token(token: str) -> dict:
    try:
        claims = jwt.decode(token, JWKS, claims_options={
            "aud": {"essential": True, "value": EXPECTED_AUDIENCE},
            "exp": {"essential": True},
            "iss": {"essential": True, "value": EXPECTED_ISSUER},
        })
        claims.validate()
        return claims
    except JoseError as e:
        raise HTTPException(401, f"Invalid token: {e}")
```

### Modus 2: code_review (kein Token-Passthrough an Upstream)

```bash
# Anti-Pattern: Token wird direkt an Upstream-Request weitergereicht
grep -rE "Authorization.*request\.headers|headers\[\"Authorization\"\]" src/
grep -rE "client_token|user_token" src/ | grep -E "fetch|httpx|requests"
```

**Fail-Pattern:**

```python
# FAIL: Token aus Client-Request landet direkt im Upstream-Call
@mcp.tool()
async def fetch_user_data(user_id: str, ctx: Context):
    token = ctx.request.headers["Authorization"]  # Client-Token
    response = await httpx.get(
        f"https://upstream.api/users/{user_id}",
        headers={"Authorization": token},  # ← Passthrough!
    )
```

**Pass-Pattern:**

```python
# OK: Eigenes Service-Token für Upstream, User-Identität via Custom-Header
@mcp.tool()
async def fetch_user_data(user_id: str, ctx: Context):
    claims = verify_token(ctx.request.headers["Authorization"])  # validate first
    if claims["sub"] != user_id and "admin" not in claims.get("scope", ""):
        return {"isError": True, "content": [TextContent(
            type="text", text="Unauthorized: cannot fetch other user's data"
        )]}

    # Eigenes Service-Token für Upstream
    service_token = await get_service_account_token()
    response = await httpx.get(
        f"https://upstream.api/users/{user_id}",
        headers={
            "Authorization": f"Bearer {service_token}",
            "X-Acting-On-Behalf-Of": claims["sub"],  # Audit-Trail
        },
    )
```

### Modus 3: runtime_test (Token mit falscher Audience wird abgelehnt)

```bash
# Token, das für anderen Service ausgestellt wurde
WRONG_AUD_TOKEN=$(generate_test_token --aud "https://other-service.ch")

curl -X POST https://mcp.schulamt-zuerich.ch/mcp/call \
  -H "Authorization: Bearer $WRONG_AUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' \
  -w "\nHTTP %{http_code}\n"
# Erwartet: HTTP 401
```

## Pass Criteria

- [ ] Token-Verifizierung prüft `aud`-Claim gegen erwarteten Wert
- [ ] Falsche/fehlende Audience → HTTP 401 (nicht 200 mit weiterem Versuch)
- [ ] Upstream-Calls verwenden eigenes Service-Account-Token, nicht Client-Token
- [ ] User-Identität wird via separatem Header (`X-Acting-On-Behalf-Of` o.ä.) propagiert für Audit-Trail
- [ ] `iss`-Claim wird ebenfalls validiert (Token-Provenienz)

## Common Failures

| Anti-Pattern | Konsequenz |
|---|---|
| Token wird mit `request.headers["Authorization"]` direkt weitergereicht | Klassisches Token-Passthrough |
| Audience wird gar nicht validiert | Token aus anderer Anwendung wird akzeptiert |
| Audience als String-Substring-Match (`"mcp" in aud`) | Token mit `audience="evil-mcp"` wird akzeptiert |
| Service-Token im Code hardcoded | Siehe ARCH-005 — separater Befund |
| Kein `X-Acting-On-Behalf-Of`-Header | Forensik unmöglich |

## Remediation

### Schritt 1: Token-Verifizierung mit Audience-Check

```python
from authlib.jose import jwt
from authlib.jose.errors import JoseError
from fastmcp import FastMCP, Context

EXPECTED_AUDIENCE = os.environ["MCP_AUDIENCE"]
EXPECTED_ISSUER = os.environ["MCP_ISSUER"]

def verify_client_token(authorization_header: str) -> dict:
    if not authorization_header.startswith("Bearer "):
        raise PermissionError("Bearer token required")
    token = authorization_header.removeprefix("Bearer ").strip()
    try:
        claims = jwt.decode(token, JWKS, claims_options={
            "aud": {"essential": True, "value": EXPECTED_AUDIENCE},
            "iss": {"essential": True, "value": EXPECTED_ISSUER},
            "exp": {"essential": True},
        })
        claims.validate()
        return dict(claims)
    except JoseError as e:
        raise PermissionError(f"Invalid token: {e}")
```

### Schritt 2: Upstream-Calls mit eigenem Service-Token

```python
from cachetools import TTLCache
import asyncio

_service_token_cache: TTLCache = TTLCache(maxsize=1, ttl=3000)
_token_lock = asyncio.Lock()

async def get_service_account_token() -> str:
    async with _token_lock:
        if "token" in _service_token_cache:
            return _service_token_cache["token"]
        # Client Credentials Grant für Service-Account
        response = await httpx.post(
            f"{OAUTH_PROVIDER}/oauth/token",
            data={
                "grant_type": "client_credentials",
                "client_id": os.environ["SVC_CLIENT_ID"],
                "client_secret": os.environ["SVC_CLIENT_SECRET"],
                "audience": "https://upstream.api",
                "scope": "data:read",  # Least-Privilege siehe SEC-003
            },
        )
        response.raise_for_status()
        token = response.json()["access_token"]
        _service_token_cache["token"] = token
        return token
```

### Schritt 3: On-Behalf-Of-Header für Audit

```python
@mcp.tool()
async def fetch_data(query: str, ctx: Context):
    user_claims = verify_client_token(ctx.request.headers["Authorization"])
    service_token = await get_service_account_token()

    response = await httpx.get(
        f"{UPSTREAM_BASE}/data?q={quote(query)}",
        headers={
            "Authorization": f"Bearer {service_token}",
            "X-Acting-On-Behalf-Of": user_claims["sub"],
            "X-MCP-Server": "schulamt-mcp/1.0",
            "X-Request-ID": str(uuid.uuid4()),  # Correlation-ID
        },
    )
```
