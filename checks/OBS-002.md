# OBS-002 — Mask Error Details

## Description

Wenn Tool-Errors Stacktraces, SQL-Syntax, Datei-Pfade oder gar Credentials enthalten, fliesst dieser Inhalt in den LLM-Kontext und damit potentiell ins User-Sichtbare zurück. Das ist Information Disclosure: Angreifer mit User-Zugriff erfahren über provozierte Errors die Server-Architektur, DB-Schema, gemountete Pfade, sogar geleakte Tokens (z.B. in `Authorization`-Headern, die im Stacktrace landen).

FastMCP bietet `mask_error_details=True`: Server-Errors werden auf eine generische Message reduziert (`"An error occurred"`), Original-Details landen nur im Server-Log. Trade-off: LLM kann nicht mehr aus Error-Details ableiten, was zu tun ist (Stichwort OBS-001 Execution-Errors). Lösung: für Execution-Errors (wo das LLM helfen soll) explizit User-Friendly-Messages, für Server-Crashes generisch maskieren.

## Verification

### Modus 1: code_review (mask_error_details aktiv)

```bash
grep -rE 'mask_error_details|maskErrors|sanitize.*error' src/
grep -rE 'traceback|format_exc|sys\.exc_info' src/
```

**Pass-Pattern:**

```python
from fastmcp import FastMCP

mcp = FastMCP(
    "server",
    mask_error_details=True,  # generic messages für unhandled errors
)

@mcp.tool()
async def query(sql: str) -> dict:
    try:
        return await db.execute(sql)
    except SQLSyntaxError as e:
        # Execution-Error: User-Friendly, ohne SQL-Internals
        return {
            "isError": True,
            "content": [TextContent(
                type="text",
                text="SQL syntax error in query. Check column names and quoting."
            )],
        }
    except Exception:
        logger.exception("Unhandled error in query tool")
        # Wird durch mask_error_details auf generische Message reduziert
        raise
```

**Fail-Pattern:**

```python
async def query(sql: str):
    try:
        return await db.execute(sql)
    except Exception as e:
        return {"error": str(e), "traceback": traceback.format_exc()}  # ← LEAKS
```

### Modus 2: runtime_test (provozierter Error)

```bash
curl -X POST https://server/mcp/call \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{
    "name":"query","arguments":{"sql":"SELECT * FROM nonexistent_table"}
  }}'
# Erwartet: User-Friendly-Message, KEIN Stacktrace, KEIN Tabellennamen-Leak
```

## Pass Criteria

- [ ] FastMCP-Init hat `mask_error_details=True` (oder TypeScript-Äquivalent)
- [ ] Keine `traceback.format_exc()` / `sys.exc_info()`-Outputs in Tool-Returns
- [ ] Execution-Errors haben User-Friendly-Messages ohne Internals
- [ ] Originale Errors landen ausschliesslich im Server-Log (nicht im Tool-Result)
- [ ] Logs sind selbst geschützt: kein Klartext-Token, kein PII

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| `return {"error": str(exc)}` ohne Sanitization | Stacktraces im LLM-Kontext |
| Generic `Exception`-Catch retourniert raw error | DB-Schema, Pfade leaken |
| FastMCP-Default ohne `mask_error_details` | Default-Verhalten je nach Version |
| Logs mit `f"{request}"` enthalten Authorization-Header | Token-Leak in Log-Aggregator |

## Remediation

```diff
  mcp = FastMCP(
      "server",
+     mask_error_details=True,
  )

  @mcp.tool()
  async def search(query: str):
      try:
          return await db.search(query)
-     except Exception as e:
-         return {"error": str(e), "traceback": traceback.format_exc()}
+     except UserInputError as e:
+         return {"isError": True, "content": [
+             TextContent(type="text", text=f"Invalid input: {e.user_message}")
+         ]}
+     except Exception:
+         logger.exception("Unhandled error in search tool")
+         raise  # mask_error_details greift, generische Message ans LLM
```
