# OBS-004 — stderr für stdio-Server

## Description

Bei stdio-Transport ist stdout **ausschliesslich** für den JSON-RPC-Protokoll-Stream reserviert. Jedes `print()`, jedes ungeleitete Logging, jeder Debug-Output an stdout korrumpiert den Stream — der Client erhält malformed JSON, bricht die Verbindung ab, der MCP-Server scheint «kaputt» zu sein.

Das Symptom ist tückisch: in lokalen Tests via Inspector funktioniert oft alles, weil der Inspector tolerant gegenüber Junk-Lines vor dem ersten valid JSON ist. In Claude Desktop bricht die Verbindung dann beim ersten `print()`.

Lösung: ALLE Logging-Outputs müssen an stderr. Stdlib `logging` muss explizit auf `sys.stderr` konfiguriert werden, structlog ebenfalls. `print()`-Statements müssen aus dem Code entfernt werden — nicht nur in Tools, auch in Dependency-Imports.

Dieser Check ist `critical`, weil er den Server unmittelbar nicht-funktional macht.

## Verification

### Modus 1: automated (print-Statements suchen)

```bash
# print()-Statements
grep -rE 'print\(' src/ --include='*.py'

# Logging-Stream-Konfiguration
grep -rE 'StreamHandler|sys\.stderr|sys\.stdout' src/

# Console-Logs in TS
grep -rE 'console\.log|console\.info' src/ --include='*.ts'
```

### Modus 2: code_review (stderr-Konfiguration)

**Pass-Pattern (Python):**

```python
import sys
import logging

logging.basicConfig(
    stream=sys.stderr,  # NICHT sys.stdout!
    level=logging.INFO,
    format="%(asctime)s %(name)s %(levelname)s: %(message)s",
)

# Mit structlog:
import structlog
structlog.configure(
    logger_factory=structlog.WriteLoggerFactory(file=sys.stderr),
    # ...
)
```

**Pass-Pattern (TypeScript):**

```typescript
import pino from "pino";

const logger = pino({
  level: "info",
  transport: {
    target: "pino/file",
    options: { destination: 2 },  // 2 = stderr
  },
});

// console.log MUSS umgeleitet werden
console.log = console.error;  // erste Verteidigungslinie für Dependencies
```

**Fail-Pattern:**

```python
print(f"Server starting on {port}")  # ← stdout, korrumpiert Protocol
logging.basicConfig()  # Default-Stream ist stderr — OK, aber fragil
```

### Modus 3: runtime_test (MCP Inspector ohne Parse-Errors)

```bash
npx @modelcontextprotocol/inspector path/to/server.py
# Erwartet: Tools werden geladen, keine "JSON parse error"-Logs
```

## Pass Criteria

- [ ] **Keine** `print()`-Statements im src/-Verzeichnis (TypeScript: kein `console.log`)
- [ ] Logging-Handler explizit auf `sys.stderr` (Python) / `destination: 2` (Node)
- [ ] structlog mit `WriteLoggerFactory(file=sys.stderr)` konfiguriert
- [ ] Test mit MCP Inspector: keine «JSON parse error»-Hinweise
- [ ] Bei dual-Transport: gleiche Logging-Konfig läuft, kein transport-abhängiger Switch

## Common Failures

| Anti-Pattern | Konsequenz |
|---|---|
| `print("Debug: ...")` für Schnell-Test vergessen | Server ab erstem Tool-Call kaputt |
| `logging.basicConfig(stream=sys.stdout)` | gleicher Effekt |
| Dependency mit eigenem `print()` (z.B. ältere `huggingface_hub`-Versionen) | unsichtbarer Bug |
| `console.log` in TypeScript ohne Override | gleicher Effekt |

## Remediation

```diff
  import sys
  import logging
+ import structlog

  logging.basicConfig(
+     stream=sys.stderr,
      level=logging.INFO,
  )

+ structlog.configure(
+     logger_factory=structlog.WriteLoggerFactory(file=sys.stderr),
+     wrapper_class=structlog.stdlib.BoundLogger,
+ )

  @mcp.tool()
  async def search(query: str):
-     print(f"Searching: {query}")
+     logger.info("searching", query=query)
      return await db.search(query)
```

Plus CI-Check (verhindert Regression):

```yaml
- name: No print statements in src
  run: |
    if grep -rn 'print(' src/ --include='*.py'; then
      echo "::error::print() found in src/ — use logger.info() to stderr"
      exit 1
    fi
```
