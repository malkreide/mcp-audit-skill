# SEC-013 — API-Key-Storage

## Description

ARCH-005 verlangt: **keine Hardcoded Secrets im Code**. SEC-013 geht weiter: in Production reichen Plain-Text Env-Vars nicht aus. Empirisch werden 53% der OSS-MCP-Server mit langlebigen Static API Keys betrieben — die in `.env`-Dateien, Container-Images, CI-Logs, und Container-Filesystem-Dumps geleakt werden.

Die Reife-Hierarchie für Secret-Storage:

| Stufe | Mechanismus | Risiko |
|---|---|---|
| **0 — Hardcoded** | Code | Repo-Push = Leak (siehe ARCH-005) |
| **1 — Plain Env-Var** | `os.environ["KEY"]`, `.env`-File | Container-Layer-Leak, CI-Log-Leak |
| **2 — Verschlüsselte Env-Var** | sops, age, sealed-secrets | OK für Repos, im Container immer noch plain |
| **3 — Secret Manager** | Vault, AWS SM (EU), Azure KV (Schweiz/EU), GCP SM (EU) | Best Practice, mit Audit-Trail + Rotation |
| **4 — Workload Identity** | IAM-Role-Assumption ohne langlebige Keys | Höchste Reife |

Für Schulamt-Kontext: Stufe 3 ist Pflicht-Minimum für `Verwaltungsdaten` und `PII`. Stufe 1 ist akzeptabel für Public-Open-Data-Server in Demo-Umgebungen, sollte aber dokumentiert werden.

## Verification

### Modus 1: code_review (Secret-Loading-Mechanismus)

```bash
# Welcher Mechanismus wird verwendet?
grep -rE 'os\.environ|process\.env' src/ | grep -iE 'KEY|SECRET|TOKEN|PASSWORD'
grep -rE 'boto3|secretsmanager|hvac|azure-keyvault|google-cloud-secret-manager' pyproject.toml package.json
```

**Pass-Pattern (Stufe 3 — AWS Secrets Manager EU):**

```python
import boto3
import json
from functools import cache

@cache
def _secrets_client():
    return boto3.client("secretsmanager", region_name="eu-central-1")

def load_secret(name: str) -> dict:
    """Lädt Secret aus AWS Secrets Manager (EU-Frankfurt für DSG-Compliance)."""
    response = _secrets_client().get_secret_value(SecretId=name)
    return json.loads(response["SecretString"])

# Beim Server-Start
secrets = load_secret("schulamt-mcp/production")
api_key = secrets["upstream_api_key"]
```

**Pass-Pattern (Stufe 4 — Workload Identity, K8s):**

```python
# Kein API-Key nötig — Pod hat IAM-Role via ServiceAccount
import boto3

# Boto3 nutzt automatisch Pod-Identity-Token
ddb = boto3.resource("dynamodb", region_name="eu-central-1")
table = ddb.Table("schulamt-data")
# Kein API-Key, keine Long-Lived-Credentials
```

**Akzeptabel-Pattern (Stufe 1, dokumentiert):**

```python
# OK für Demo-Server mit Public Open Data
# Dokumentiert in docs/secret-management.md:
# "Server nutzt Plain Env-Vars. Akzeptabel weil:
#  - Datenklasse: Public Open Data
#  - Deployment: Demo-Railway, kein Production-Traffic
#  - API-Key der oberen Quelle ist non-sensitive (Public-API-Quote)"
import os
api_key = os.environ["UPSTREAM_API_KEY"]
```

### Modus 2: config_check (Production-Deployment)

```bash
# Container-Image enthält keine Secrets
docker history <image> --no-trunc | grep -iE 'KEY|SECRET|TOKEN'

# Deployment-Manifest referenziert Secret-Manager
grep -rE 'secretKeyRef|valueFrom.*secret|external.secret' k8s/ helm/
```

**Pass-Pattern (Kubernetes mit External Secrets Operator):**

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: mcp-server-secrets
spec:
  secretStoreRef:
    name: aws-secrets-eu
    kind: ClusterSecretStore
  target:
    name: mcp-server-secrets
  data:
    - secretKey: API_KEY
      remoteRef:
        key: schulamt-mcp/production
        property: upstream_api_key
---
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: mcp-server
          envFrom:
            - secretRef:
                name: mcp-server-secrets
```

### Modus 3: code_review (Rotation-Support)

Secrets müssen ohne Code-Änderung rotierbar sein.

```bash
grep -rE 'lru_cache|cache_ttl|refresh.*secret|reload.*config' src/
```

**Pass:** Server lädt Secrets entweder bei jedem Tool-Call neu (kostspielig, sicher) oder mit TTL-Cache (z.B. 5-15 min) für Reload nach Rotation.

```python
from cachetools import TTLCache

_secret_cache = TTLCache(maxsize=10, ttl=300)  # 5min TTL

def get_api_key() -> str:
    if "api_key" not in _secret_cache:
        _secret_cache["api_key"] = load_secret("schulamt-mcp/production")["api_key"]
    return _secret_cache["api_key"]
```

## Pass Criteria

- [ ] Bei `Verwaltungsdaten`/`PII`: Stufe 3 (Secret Manager) oder 4 (Workload Identity)
- [ ] Bei `Public Open Data`: Stufe 1 ist akzeptabel, aber dokumentiert in `docs/secret-management.md`
- [ ] Container-Image enthält keine Secrets im Layer (`docker history`-Test bestanden)
- [ ] Secret-Manager-Region ist Schweiz/EU (DSG, siehe CH-001)
- [ ] Rotation ist möglich ohne Code-Änderung (TTL-Cache oder Reload-Mechanismus)
- [ ] Secret-Access wird auditiert (Secret-Manager-eigenes Audit-Log oder OBS-005)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Plain Env-Var in Container-Image-`ENV`-Layer | Image-Pull = Secret-Leak |
| `.env`-File in Repo (auch wenn `.gitignore`) | Versehen-Push, History-Leak |
| Secrets-Manager in USA-Region | DSG-Verstoss bei nicht-public Datenklasse |
| Long-Lived API-Keys ohne Rotation-Verfahren | Bei Leak: kein Recovery-Pfad |
| Secret-Cache ohne TTL — ewig im Memory | Rotation greift erst bei Pod-Restart |

## Remediation

### Schritt 1: Reife-Stufe pro Server bestimmen

| Server-Profil | Empfohlene Stufe |
|---|---|
| `Public Open Data`, Demo-Deployment | 1 (dokumentiert) |
| `Public Open Data`, Production | 3 |
| `Verwaltungsdaten` | 3 oder 4 |
| `PII` | 4 (Workload Identity) bevorzugt |

### Schritt 2: Migration zu Secret Manager (Stufe 1 → 3)

```diff
- import os
- api_key = os.environ["UPSTREAM_API_KEY"]

+ import boto3, json
+ from cachetools import TTLCache
+
+ _cache = TTLCache(maxsize=1, ttl=300)
+
+ def get_api_key() -> str:
+     if "key" not in _cache:
+         client = boto3.client("secretsmanager", region_name="eu-central-1")
+         response = client.get_secret_value(SecretId="schulamt-mcp/production")
+         _cache["key"] = json.loads(response["SecretString"])["upstream_api_key"]
+     return _cache["key"]
```

### Schritt 3: Rotation-Verfahren dokumentieren

`docs/secret-rotation.md`:

```markdown
# Secret Rotation Procedure

## Trigger
- Vermutung eines Leaks
- Mitarbeiter-Wechsel mit Secret-Zugriff
- Routine: alle 90 Tage

## Schritte
1. Neuen Secret im AWS Secrets Manager erzeugen (versioned)
2. Production-Server warten 5min (TTL-Cache läuft ab)
3. Verifizieren: erfolgreiche Tool-Calls mit neuer Version
4. Alten Secret nach 24h löschen
```

### Schritt 4: Workload Identity (Stufe 4) — falls möglich

Bei Kubernetes-Deployment mit Cloud-Provider-Integration: ServiceAccount-Token statt langlebige Keys.

## Effort

M — 1–3 Tage. Secret-Manager-Setup + Migrations-Skript + Rotation-Doku.
