# OBS-001 — Protocol vs. Execution Errors

## Description

Die MCP-Spezifikation fordert eine strikte Trennung zwischen zwei Fehler-Typen. Werden sie verwechselt, kann das LLM den Fehler nicht korrekt interpretieren und bricht in eine Halluzinations- oder Sackgassen-Schleife.

| Fehler-Typ | Beispiele | Format |
|---|---|---|
| **Protocol Error** | Tool existiert nicht, Schema-Mismatch, JSON-Parsing-Fehler, interner Server-Crash beim Routing | Standard JSON-RPC Error Response (`{"jsonrpc": "2.0", "error": {...}}`) |
| **Execution Error** | API-Ratenlimit, Datei nicht gefunden, ungültige Geschäfts-Parameter, Drittanbieter-API down | Tool-Result mit `isError: true`, **niemals** als JSON-RPC-Error |

**Warum das wichtig ist:** Wenn ein Datei-Lookup-Tool «Datei nicht gefunden» als JSON-RPC-Error wirft, denkt das LLM: «Verbindung zum Tool gebrochen, kann es nicht mehr verwenden». Wenn es als Execution Error mit `isError: true` und Text «File not found: /var/log/app.log» kommt, denkt das LLM: «Pfad ist falsch, ich versuche `find /var -name app.log` mit dem List-Tool».

## Verification

### Modus 1: code_review (Tool-Handler-Pattern)

```bash
# Suche nach Exception-Handling in Tool-Handlern
grep -rB5 -A20 "@mcp\.tool\|@server\.tool" src/ | grep -E "raise|throw|except|catch"
```

**Pass-Pattern (Python / FastMCP):**

```python
from mcp.types import TextContent
from mcp.server.fastmcp import Context

@mcp.tool()
async def read_log(ctx: Context, path: str) -> dict:
    try:
        content = await asyncio.to_thread(open(path).read)
        return {"content": content, "path": path}
    except FileNotFoundError:
        # Execution Error: nicht raisen, sondern als isError zurückgeben
        return {
            "isError": True,
            "content": [
                TextContent(
                    type="text",
                    text=f"File not found: {path}. Try listing the parent directory first."
                )
            ],
        }
    except PermissionError:
        return {
            "isError": True,
            "content": [
                TextContent(type="text", text=f"Permission denied for {path}.")
            ],
        }
    # Protocol Errors (z.B. interne Crashes) bubbeln durch und werden vom Framework
    # als JSON-RPC-Error formatiert
```

**Fail-Pattern:**

```python
@mcp.tool()
async def read_log(path: str) -> str:
    return open(path).read()  # FileNotFoundError → Framework wirft JSON-RPC Error
```

### Modus 2: code_review (Fehlercode-Konstanten)

```bash
# Suche nach standardisierten Fehlercodes
grep -rE "INVALID_TOOL|INVALID_PARAMS|INSUFFICIENT_SCOPE|RATE_LIMIT_EXCEEDED|READ_ONLY_MODE|EXECUTION_FAILED" src/
grep -rE "-32601|-32602|-32001|-32003|-32004|-32603" src/
```

**Pass:** Standardisierte Codes verwendet:

| Code | HTTP | Zweck |
|---|---|---|
| -32601 | 400 | INVALID_TOOL |
| -32602 | 400 | INVALID_PARAMS |
| -32001 | 403 | INSUFFICIENT_SCOPE |
| -32003 | 429 | RATE_LIMIT_EXCEEDED |
| -32004 | 403 | READ_ONLY_MODE |
| -32603 | 500 | EXECUTION_FAILED |

### Modus 3: runtime_test (Fehler-Verhalten)

Provokative Anfragen senden, prüfen wie der Server antwortet:

```bash
# Existierendes Tool mit invalider Eingabe → soll Execution Error werfen
curl -X POST https://server/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"read_log","arguments":{"path":"/nonexistent"}}}'
# Erwartung: result mit isError:true, NICHT error mit message:"File not found"

# Nicht existierendes Tool → soll Protocol Error werfen
curl -X POST https://server/mcp \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"nonexistent_tool"}}'
# Erwartung: error mit code:-32601 (INVALID_TOOL)
```

## Pass Criteria

- [ ] Tool-Handler fangen anwendungsspezifische Fehler (FileNotFound, RateLimit, InvalidArgument) ab
- [ ] Anwendungsfehler werden mit `isError: true` in `tool-result` zurückgegeben (nicht als JSON-RPC-Error)
- [ ] Standardisierte Fehlercodes (-326xx, -320xx) für Protocol-Level-Errors
- [ ] Mindestens 1 dokumentierter Test deckt Execution-Error-Pfad ab
- [ ] Mindestens 1 dokumentierter Test deckt Protocol-Error-Pfad ab (falsches Tool / falsche Args)

## Common Failures

| Anti-Pattern | Konsequenz |
|---|---|
| Alle Errors als Python-Exception raisen | LLM denkt Verbindung kaputt, bricht ab |
| Generisches `try/except: raise McpError` | Verlust aller Diagnose-Information |
| Custom Error-Codes ohne JSON-RPC-Standard | Clients können Errors nicht maschinell unterscheiden |
| Stack-Traces direkt im error-Body | Information Disclosure (siehe OBS-002) |

## Remediation

```diff
+ from mcp.types import TextContent
+
  @mcp.tool()
  async def query_database(query: str) -> dict:
-     # FAIL: alle Exceptions werden zu JSON-RPC-Errors
-     conn = await asyncpg.connect(DATABASE_URL)
-     return {"rows": await conn.fetch(query)}
+     try:
+         conn = await asyncpg.connect(DATABASE_URL)
+         try:
+             rows = await conn.fetch(query)
+             return {"rows": [dict(r) for r in rows]}
+         finally:
+             await conn.close()
+     except asyncpg.PostgresSyntaxError as e:
+         # Execution Error: Query-Problem ist Aufgabe des LLMs zu lösen
+         return {
+             "isError": True,
+             "content": [TextContent(
+                 type="text",
+                 text=f"SQL syntax error: {str(e)}. Try simplifying the query."
+             )],
+         }
+     except asyncpg.PostgresConnectionError:
+         # Protocol-nahe: Server ist degraded
+         raise McpError(code=-32603, message="Database temporarily unavailable")
```

## Effort

M — 1–3 Tage. Pro Tool muss der Error-Pfad reviewed werden. Bei vielen Tools (>10) entsprechend aufwändiger.
