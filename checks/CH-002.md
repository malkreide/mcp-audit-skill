# CH-002 — DSG-konforme Personendaten-Verarbeitung

## Description

Das revidierte Schweizer Datenschutzgesetz (revDSG, in Kraft seit 1.9.2023) verlangt für jede Verarbeitung von Personendaten eine **explizite Rechtsgrundlage** (Art. 31). Im öffentlichen Sektor (Art. 34) sind die zulässigen Grundlagen typischerweise:

1. **Gesetzliche Grundlage** — die Verarbeitung ist in einem formellen Gesetz vorgesehen (z.B. Volksschulgesetz Kanton Zürich)
2. **Aufgabenerfüllung** — die Verarbeitung ist nötig zur Erfüllung der gesetzlichen Aufgabe
3. **Einwilligung** — die betroffene Person hat ausdrücklich zugestimmt
4. **Lebenswichtiges Interesse** — eng begrenzt, kaum für MCP-Server relevant
5. **Öffentliches Interesse** — bei einer im Gesetz definierten Aufgabe

Für einen MCP-Server der Stadt Zürich, der Personendaten verarbeitet (z.B. Schüler-IDs, Lehrer-Namen, Eltern-Kontakte), muss **vor dem ersten Produktiv-Einsatz** dokumentiert sein:

- Welche Personendaten werden verarbeitet?
- Auf welcher Rechtsgrundlage?
- Welche Zwecke werden verfolgt?
- Welche Datenkategorien (insb. besonders schützenswerte Daten nach Art. 5 lit. c)?
- Wer sind die Empfänger?
- Wie lange werden Daten aufbewahrt?

Dieser Check ist `critical`, weil ohne Rechtsgrundlage jede einzelne Verarbeitung eine DSG-Verletzung darstellt — mit Bussen-, Reputations- und Persönlichkeitsschutz-Folgen.

## Verification

### Modus 1: documentation_check (Verarbeitungsverzeichnis)

```bash
find . -iname 'verarbeitungsverzeichnis*' -o -iname 'processing-record*' \
       -o -iname 'dsg*' -o -iname 'data-protection*' -o -iname 'rechtsgrundlage*'
ls docs/ -la
```

**Pass:** Mindestens eine Datei mit Verarbeitungsverzeichnis im Repo oder verlinkt aus dem README. Inhalt muss enthalten:

- Bezeichnung der Verarbeitung
- Verantwortlicher (z.B. Schulamt Stadt Zürich)
- Zweck
- Rechtsgrundlage (mit Verweis auf konkretes Gesetz/Artikel)
- Kategorien Personendaten
- Kategorien betroffener Personen
- Datenempfänger (intern/extern)
- Aufbewahrungsdauer
- Sicherheitsmassnahmen

### Modus 2: code_review (Datenfluss überprüfbar)

```bash
# Welche Personendaten-Felder werden vom Server verarbeitet?
grep -rE 'name|email|address|birthdate|phone|ahv|user_id|schueler' src/ --include='*.py'
```

Der Maintainer muss anhand des Codes belegen können:

- Welche Felder als PII gelten
- Wo sie eintreten (welche API/Quelle)
- Wie sie verarbeitet werden
- Ob sie weitergeleitet werden (an LLM via Sampling? an externe APIs?)

### Modus 3: documentation_check (Hinweis im README)

```bash
grep -iE 'datenschutz|dsg|rechtsgrundlage|personendaten' README.md README.de.md
```

**Pass:** README enthält Sektion mit Verweis auf Verarbeitungsverzeichnis und Rechtsgrundlage.

### Modus 4: documentation_check (Bei Einwilligung: Einwilligungs-Mechanismus)

Falls Rechtsgrundlage = Einwilligung, muss dokumentiert sein:

- Wie die Einwilligung eingeholt wird (UI-Flow, Schriftlich)
- Wie sie zurückziehbar ist (Art. 31 Abs. 1)
- Wie sie nachgewiesen wird (Audit-Log mit Zeitstempel)

## Pass Criteria

- [ ] Verarbeitungsverzeichnis im Repo vorhanden (`docs/dsg-processing-record.md` oder ähnlich)
- [ ] Rechtsgrundlage explizit benannt (mit Gesetz und Artikel)
- [ ] Aufbewahrungsdauer dokumentiert
- [ ] Datenempfänger dokumentiert
- [ ] README verweist auf Verarbeitungsverzeichnis
- [ ] Bei Einwilligung: Einwilligungs- und Widerrufs-Mechanismus dokumentiert
- [ ] Bei besonders schützenswerten Daten (Art. 5 lit. c — Gesundheit, religiöse Ansichten, etc.): zusätzliche Begründung dokumentiert

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Kein Verarbeitungsverzeichnis | DSG-Pflichtverletzung, EDÖB-relevant |
| «Datenverarbeitung im Rahmen unserer Tätigkeit» — kein konkretes Gesetz | Ungenügende Rechtsgrundlage |
| Aufbewahrungsdauer «bis auf Widerruf» | DSG-konform nur in eng begrenzten Fällen |
| Empfängerliste vergisst LLM-Provider (z.B. Anthropic via Sampling) | Verstecker Drittlandtransfer |
| Einwilligung von Schülern unter 16 ohne Eltern-Einwilligung | Art. 31 Abs. 1 verletzt |

## Remediation

### Schritt 1: Verarbeitungsverzeichnis erstellen

Template in `docs/dsg-processing-record.md`:

```markdown
# Verarbeitungsverzeichnis: <server-name>

## Verantwortlicher
Schulamt Stadt Zürich, Parkring 4, 8027 Zürich
Datenschutz-Kontakt: [datenschutz@zuerich.ch]

## Bezeichnung der Verarbeitung
<Klare, kurze Beschreibung — z.B. "Suche in Bildungsstatistiken">

## Zweck
<Konkret — z.B. "Politische Berichterstattung, interne Reportings">

## Rechtsgrundlage
- [ ] Gesetzliche Grundlage: <Artikel und Gesetz>
- [ ] Aufgabenerfüllung: <Welche Aufgabe?>
- [ ] Einwilligung der betroffenen Person
- [ ] Andere: <Erläuterung>

## Kategorien Personendaten
- z.B. Vorname, Nachname, Schüler-ID, Klassen-Zuordnung
- (Falls besonders schützenswert nach Art. 5 lit. c): <markieren>

## Kategorien betroffener Personen
- z.B. Schüler:innen der Volksschule Stadt Zürich (ca. 32'000)
- z.B. Lehrpersonen (ca. 4'500)

## Datenempfänger
- Intern: <Liste städtischer Stellen>
- Extern: <Liste mit Rechtsgrundlage und Vertrag>
- LLM-Provider (falls Sampling verwendet): <Anthropic, EU-Region>

## Aufbewahrungsdauer
<Konkret in Monaten/Jahren, mit Begründung>

## Löschkonzept
<Wie und wann werden Daten gelöscht?>

## Sicherheitsmassnahmen
- Datenresidenz: <Schweiz / EU>
- Verschlüsselung at-rest: <ja/nein, wie>
- Verschlüsselung in-transit: <TLS-Version>
- Zugriffskontrolle: <RBAC-Modell>
- Audit-Logs: <Aufbewahrung, Schutz>

## Datenübermittlungen ins Ausland
<Falls relevant: SCC, BCRs, Einwilligung>

## Datum letzte Aktualisierung
YYYY-MM-DD
```

### Schritt 2: README-Sektion ergänzen

```markdown
## Datenschutz

Dieser MCP-Server verarbeitet Personendaten der Stadt Zürich.
Das Verarbeitungsverzeichnis nach revDSG Art. 12 ist in
[docs/dsg-processing-record.md](./docs/dsg-processing-record.md) dokumentiert.

Rechtsgrundlage: <Gesetz und Artikel>.
```

### Schritt 3: Beim Datenschutzbeauftragten der Stadt Zürich vorlegen

Verarbeitungsverzeichnis ist nicht intern zu führen — es muss bei der städtischen Datenschutzaufsicht eingereicht und dokumentiert sein.

## Effort

L — 1–2 Wochen pro Server. Verarbeitungsverzeichnis erstellen + Review intern + Sign-off Datenschutzbeauftragte:r.
