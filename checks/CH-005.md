# CH-005 — ISDS Schutzbedarfsklasse-Mapping

## Description

Die Stadt Zürich verwendet die ISDS-Richtlinie (Informationssicherheit & Datenschutz) zur Klassifikation aller Informationen und IT-Systeme. Die Schutzbedarfsklassen unterscheiden sich nach den drei Schutzzielen Vertraulichkeit, Integrität und Verfügbarkeit, wobei jede der drei Dimensionen separat klassifiziert wird:

| Klasse | Vertraulichkeit | Integrität | Verfügbarkeit |
|---|---|---|---|
| **Niedrig** | Öffentlich | Geringe Schäden bei Manipulation | Stillstand bis 7 Tage tolerierbar |
| **Mittel** | Intern (Stadtverwaltung) | Mittlere Schäden | Stillstand max 24h |
| **Hoch** | Vertraulich | Wesentliche Schäden | Stillstand max 4h |
| **Sehr hoch** | Streng vertraulich | Existenzgefährdend | < 1h |

Für jeden MCP-Server der Stadt Zürich muss vor der Inbetriebnahme:

1. Pro Schutzziel die Klasse bestimmt sein (3 separate Bewertungen)
2. Die Bewertungen begründet dokumentiert sein
3. Sicherheitsmassnahmen entsprechend der höchsten Klasse umgesetzt sein
4. Bei Klasse «Hoch» oder «Sehr hoch»: Genehmigung durch OIZ-Sicherheit eingeholt sein

Beispiel `zh-education-mcp`: Vertraulichkeit «Niedrig» (Public Open Data), Integrität «Mittel» (falsche Statistiken könnten politische Entscheidungen beeinflussen), Verfügbarkeit «Niedrig» (Demo-Server, kein 24/7-Betrieb nötig). Massnahmen entsprechend «Mittel».

## Verification

### Modus 1: documentation_check (ISDS-Klassifikation im Repo)

```bash
find . -iname '*isds*' -o -iname '*schutzbedarf*' -o -iname '*klassifikation*'
grep -iE 'isds|schutzbedarf|vertraulich|integrität|verfügbarkeit' README.md docs/
```

**Pass-Pattern (`docs/isds-klassifikation.md`):**

```markdown
# ISDS-Klassifikation: <server-name>

## Schutzbedarfsklasse

| Schutzziel | Klasse | Begründung |
|---|---|---|
| Vertraulichkeit | <Niedrig/Mittel/Hoch/Sehr hoch> | <Begründung> |
| Integrität | <Niedrig/Mittel/Hoch/Sehr hoch> | <Begründung> |
| Verfügbarkeit | <Niedrig/Mittel/Hoch/Sehr hoch> | <Begründung> |

## Höchste Klasse: <Klasse>
Sicherheitsmassnahmen werden gemäss dieser Klasse umgesetzt.

## Genehmigung (bei Hoch / Sehr hoch)
- OIZ-Sicherheit-Sign-off: <Datum, Person>
- Datenschutzbeauftragte:r-Sign-off: <Datum, Person>

## Massnahmen-Mapping
- Vertraulichkeit <Klasse>: <konkrete Massnahmen>
- Integrität <Klasse>: <konkrete Massnahmen>
- Verfügbarkeit <Klasse>: <konkrete Massnahmen>
```

### Modus 2: code_review (Massnahmen entsprechen Klasse)

Pro Klassen-Stufe werden konkrete Massnahmen verlangt. Beispiele:

| Klasse | Vertraulichkeit-Massnahmen | Integritäts-Massnahmen | Verfügbarkeits-Massnahmen |
|---|---|---|---|
| Mittel | TLS, Auth, Audit-Logs | Input-Validation, Schema-Checks | Health-Checks, Restart-Policy |
| Hoch | + Verschlüsselung at-rest, RBAC | + Cryptographic Signatures, Tamper-Detection | + Multi-Region, < 4h MTTR |
| Sehr hoch | + Air-Gap / dedizierte Infra | + HSM, Multi-Sig | + Active-Active, < 1h MTTR |

### Modus 3: documentation_check (Genehmigung bei Hoch/Sehr hoch)

```bash
grep -iE 'oiz|sign-off|genehmigung|approved' docs/ README.md
```

**Pass:** Bei Klasse «Hoch» oder höher: Sign-off-Datum und verantwortliche Person dokumentiert.

## Pass Criteria

- [ ] ISDS-Klassifikations-Dokument im Repo (`docs/isds-klassifikation.md`)
- [ ] Pro Schutzziel separate Bewertung (Vertraulichkeit, Integrität, Verfügbarkeit)
- [ ] Bewertung mit konkreten Begründungen
- [ ] Massnahmen-Mapping pro Klasse dokumentiert
- [ ] Bei Klasse «Hoch» oder «Sehr hoch»: Genehmigung durch OIZ-Sicherheit dokumentiert
- [ ] Bei PII-Verarbeitung: Datenschutzbeauftragte:r-Sign-off (Synergie zu CH-002)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Globale Klasse «Mittel» ohne Aufschlüsselung pro Schutzziel | Massnahmen können einzelne Dimensionen unterversorgen |
| Keine ISDS-Klassifikation | Compliance-Verstoss bei Stadt-Zürich-Audit |
| Klasse «Hoch» ohne OIZ-Sign-off | Unauthorized Production-Setup |
| Massnahmen nur für Vertraulichkeit, keine für Integrität/Verfügbarkeit | Unvollständige Risikoabdeckung |

## Remediation

### Schritt 1: Klassifikations-Workshop

Mit Maintainer + Datenschutzbeauftragte:r + ggf. KI-Fachgruppe:

1. Pro Schutzziel die typischen Schadensszenarien durchgehen
2. Klasse zuordnen (Niedrig/Mittel/Hoch/Sehr hoch)
3. Begründungen dokumentieren
4. Höchste Klasse als Massnahmen-Massstab nehmen

### Schritt 2: Massnahmen-Mapping

Pro Klasse vorhandene Massnahmen dokumentieren und Lücken identifizieren. Lücken als Findings in den Audit-Tracker.

### Schritt 3: Bei Hoch/Sehr hoch: OIZ-Beauftragung

OIZ-Sicherheit-Kontakt aufnehmen, Klassifikations-Dokument einreichen, Sign-off einholen.

## Effort

M — 1–2 Tage für Klassifikation und Doku. Bei Klasse Hoch zusätzlich 1–2 Wochen für OIZ-Genehmigungs-Prozess.
