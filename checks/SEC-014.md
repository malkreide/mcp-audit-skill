# SEC-014 — Tool-Allow-Listing via MCP-Gateway

## Description

Wenn mehrere MCP-Server in einer Organisation laufen, exponiert jeder Server Dutzende von Tools. Der LLM-Client (z.B. Claude Desktop pro Mitarbeiter) erhält die volle Tool-Manifest-Sammlung — und kann jedes Tool aufrufen. Ohne zentrale Kontrolle entstehen drei Risiken:

1. **Lateral Movement zwischen Servern:** Ein kompromittierter Server kann den LLM dazu bringen, Tools eines anderen Servers aufzurufen — z.B. Daten aus `schulamt-mcp` exportieren via Tool von `gateway-mcp`.

2. **Rolle-/Team-übergreifende Tool-Aufrufe:** Ein Schulamt-Mitarbeiter sollte nicht versehentlich Tools des Finanz-MCPs aufrufen können, auch wenn beide via OAuth korrekt authentifiziert sind.

3. **Compliance-Verstösse durch Tool-Kombinationen:** Tool A liefert Verwaltungsdaten, Tool B sendet E-Mails an Externe. Die Kombination beider durch das LLM erzeugt einen Datenfluss, der einzeln okay, kombiniert aber meldepflichtig ist.

Mitigation: **MCP-Gateway als zentrale Vermittlung** (verwandt zu SCALE-005). Pro Team / Rolle wird eine Allow-List definiert: welche Tools welcher Server darf der LLM aufrufen? Tools ausserhalb der Allow-List werden vor der Tool-Liste vom Gateway gefiltert — der LLM sieht sie gar nicht.

## Verification

### Modus 1: documentation_check (Allow-List konfiguriert)

```bash
find . -iname '*allowlist*' -o -iname '*tool-policy*' -o -iname '*gateway-config*'
grep -rE 'allowed_tools|tool_allowlist|whitelist' deploy/ config/
```

**Pass-Pattern (`gateway-config/schulamt-team.yaml`):**

```yaml
# MCP Gateway Tool Allow-List für Schulamt-Team
team: schulamt
team_oauth_groups:
  - "schulamt-mitarbeiter"
  - "schulamt-leitung"

allowed_servers:
  zh-education-mcp:
    allowed_tools:
      - search_education_stats
      - get_school_info
      - list_districts
    denied_tools: []  # explizit leer für Audit-Klarheit

  parlament-mcp:
    allowed_tools:
      - search_motions
      - get_motion_details

  bag-health-mcp:
    # Komplett gesperrt — Schulamt braucht keine Health-Daten
    allowed_tools: []

denied_servers:
  - register-mcp     # PII-Server, separater Zugang via Spezialrolle
  - bag-epl-mcp      # FHIR-Daten, nicht für Schulamt-Standard

audit:
  log_all_calls: true
  alert_on_denied: true
```

### Modus 2: code_review (Server-Side Allow-List-Check)

Der Gateway selbst prüft das vor dem Tool-Forward. Aber: Defense-in-Depth verlangt, dass auch der Server eine grobe Validation hat — z.B. via OAuth-Group-Membership.

```bash
grep -rE 'group|role|team' src/ | grep -E 'check|validate|require'
```

**Pass-Pattern (Server-side Group-Check, ergänzend):**

```python
def require_group(*allowed_groups: str):
    def decorator(tool_func):
        @functools.wraps(tool_func)
        async def wrapper(args, ctx: Context):
            claims = ctx.user_claims  # aus SEC-002
            user_groups = set(claims.get("groups", []))
            if not user_groups & set(allowed_groups):
                # Generischer 403 — kein Hinweis auf erlaubte Gruppen
                return {
                    "isError": True,
                    "content": [TextContent(
                        type="text", text="Tool not available."
                    )],
                }
            return await tool_func(args, ctx)
        return wrapper
    return decorator


@mcp.tool()
@require_group("schulamt-leitung", "stadtverwaltung-admin")
async def export_internal_report(args, ctx):
    return await db.export_report(args.report_id)
```

### Modus 3: runtime_test (Denied Tool wird gefiltert)

Test: User aus Schulamt-Gruppe ruft Tool auf, das nicht in Allow-List ist.

```bash
SCHULAMT_TOKEN=$(create_test_jwt --groups "schulamt-mitarbeiter")

# Tool in Allow-List
curl -X POST https://gateway/mcp/tools/list \
  -H "Authorization: Bearer $SCHULAMT_TOKEN" \
  | jq '.tools[].name'
# Erwartet: nur Tools aus Allow-List sichtbar

# Tool ausserhalb Allow-List direkt aufrufen
curl -X POST https://gateway/mcp/tools/call \
  -H "Authorization: Bearer $SCHULAMT_TOKEN" \
  -d '{"name":"register-mcp.search_persons", "arguments":{"q":"test"}}'
# Erwartet: HTTP 403 oder isError mit "Tool not available"
```

## Pass Criteria

- [ ] Bei `enterprise_context`: Tool-Allow-List pro Team/Rolle dokumentiert (im Gateway-Config oder Server-Config)
- [ ] Allow-List ist explizit ("default-deny"): nur gelistete Tools werden exponiert
- [ ] Server-Side Defense-in-Depth: Group-/Role-Check für sensitive Tools, ergänzend zum Gateway
- [ ] Denied-Tool-Aufrufe werden auditiert (siehe OBS-005)
- [ ] Tool-Liste, die der LLM-Client sieht, ist team-/rollen-spezifisch gefiltert (im `tools/list`-Response)

## Common Failures

| Anti-Pattern | Risiko |
|---|---|
| Globale Allow-List für alle User | Lateral Movement zwischen Teams möglich |
| Allow-List nur am Gateway, nicht am Server | Bei Gateway-Bypass kein Schutz |
| Default-Allow statt Default-Deny | Neu hinzugefügte Tools sind automatisch zugänglich |
| Keine Audit-Logs für denied tool calls | Forensik bei Probing-Versuchen unmöglich |

## Remediation

### Schritt 1: Tool-Inventar pro Team

Pro Team/Rolle dokumentieren, welche Use-Cases anfallen und welche Tools dafür nötig sind. Output: Allow-Listen.

### Schritt 2: Gateway-Konfiguration

YAML wie im Pass-Pattern, eingecheckt im zentralen Gateway-Config-Repo (separater Repo, eigene Audit-Hierarchie).

### Schritt 3: Server-Side `require_group`-Decorator

Wie im Pass-Pattern. Mindestens für `Verwaltungsdaten`/`PII`-Tools verpflichtend.

### Schritt 4: Audit-Logging für denied tool calls

```python
@mcp.middleware
async def log_tool_attempts(request, call_next):
    response = await call_next(request)
    if response.status_code == 403 and "tools/call" in request.url.path:
        audit_logger.warning(
            "tool_call_denied",
            user=request.state.user_claims.get("sub"),
            attempted_tool=request.json.get("name"),
            user_groups=request.state.user_claims.get("groups"),
        )
    return response
```

### Schritt 5: Periodischer Allow-List-Review

Quartalsweise prüfen: sind alle Allow-List-Einträge noch relevant? Welche User-Beschwerden gab's wegen denied calls? Allow-List zu breit oder zu eng?

## Effort

L — 1–2 Wochen für Initial-Setup. Quartalsweise ~1 Tag für Review.
